## Summary

The sync test suite is very large and unusually well-commented but structurally symptomatic. Scope: ~54 test files match *sync* (plus pairing/split/voiceover-companion adjacents), ~27k LOC total; the core engine slice (tests/slides/test_sync_* + tests/cli/test_slides_sync*) is ~45 files, ~920 test functions, ~24k LOC. Scenarios are expressed as inline f-string builders of raw jupytext percent-format cell headers (`_slide`, `_deck`, `_ncode`, `_idless_code`, ...) that are RE-DEFINED PER FILE — 39 files define local `_slide`/`_write_pair`/`_deck`, and ~15 files each carry their own `_seed_watermark`/`_seed` re-implementation of the production `_record_watermark`, in at least 3 divergent schema generations (2-partition 5-tuples, 3-partition "membership-widened" de/en/shared, and a tags=/anchors= kwarg generation). There is no shared scenario DSL, no property-based testing anywhere in the sync suite, and heavy white-box coupling: tests import ≥15 private underscore symbols from the engine (_record_watermark, _apply_ledger_overlay, _baseline_from_watermark, _conflict_as_edit, _plan_exit_code/_apply_exit_code, _outcome_line, _resolve_alignment, _maybe_retag) and one meta-test hardcodes ~28 (partition,field)→private-function-name mappings.

What the tests pin down is genuinely strong on some axes: the no-op invariant (a nothing-to-do plan applies with ZERO bytes written and ZERO LLM calls, checked over a 212-pair real corpus), the #269 "propagate-or-alert, never silently drop" cardinal invariant (formalized as a reusable `_falsely_consistent` predicate and exercised both synthetically and by a corpus mutation oracle), single-rerun idempotency (apply-then-rerun is a noop, watermark-advance idempotency), dry-run/apply parity (#216: the preview must predict the writing run's exit code and effects), per-pass write atomicity (an erroring pass flushes nothing, no stray temp files, truthful "rolled back / nothing written" reporting), both-sides-edited conflict handling (deferral, degradation to per-cell conflicts, equivalence-gate downgrade, the #447 de-wins/safe/escalate policy tiers), and byte-exactness of untouched cells. What they miss: multi-round convergence (nothing beyond one rerun — no alternating-direction edit sequences, no fixpoint property over generated decks), crash-injection between the per-file os.replace calls (multi-file atomicity is only tested as "erroring pass writes nothing in memory-buffered mode"), any fuzz/property coverage of the enormous hand-enumerated cell-class space, and the 4-file separated-companion write-back (this worktree is at #501 Phase 1: companion apply REFUSES by design and only the refusal is tested; Phase 2/PR #515 is not in this tree). The `clm voiceover sync` companion tests stub the merge pipeline (`_merge_notes` with patched `merge_batch`), so companion merging is tested only below the LLM boundary.

The fragility signals are the loudest finding. 38 distinct issue numbers are referenced 189 times across 30 sync test files; two whole files are named after incidents (test_sync_issue_269.py — 44 tests, each run under BOTH watermark and git-HEAD baselines; test_sync_idless_localized_drift_repro.py — characterization repros of the #364/#365/#366 stale-watermark incident); test_sync_limitations.py (1,491 lines) exists to document engine limitations that were later fixed, and its tail bundles "fast reproduction of the two corpus-mutation oracle failures" (#443). test_sync_plan.py enumerates ~30 cold-start refusal/reconcile cases across {watermark, git-head, no-baseline} × {id-less, id-carrying, half-id'd, mismatched, mixed} × {provider, no-provider} × {committed, uncommitted} — direct evidence of combinatorial special-casing in the classifier. The tests also reveal API smells: conflict decisions are keyed by Python `id(proposal)` (in-memory object identity), and the corpus oracles hardcode the maintainer's personal PythonCourses path as a fallback corpus root.

On the #298/#299 oracle infrastructure: the corpus no-op backstop and the corpus mutation oracle are the two real oracles, and the mutation oracle demonstrably caught real bugs (#443). But their CI strength is weak: real-corpus runs are `slow`-marked and NEVER run in CI; the CI fallback is a single 14-line synthetic deck pair, on which the scale floors are explicitly skipped. The mutation oracle covers only 6 scripted ONE-sided EN-half mutations on ≤3 pairs per cell-class; no DE-side or both-sided corpus mutations. The channel-coverage meta-test (the #289-class closer for "recorded-but-unconsumed watermark channel") is a hand-maintained registry whose verification is only `hasattr(module, name)` — it proves a named private function still exists, not that it actually consumes the channel. Phase-0 measurement embedded in the no-op test is itself damning context: only 81 of 212 real corpus pairs were post-sync-clean, and the regression floor is set at 40 (half the measured value), i.e. the engine cannot bring ~62% of the real corpus to a clean state and the guard tolerates losing half of what works.

## Key findings

- Volume: ~54 sync-matched test files, ~27k LOC, ~1,100 test functions; core engine slice = ~45 files / ~920 tests / ~24k LOC (tests/slides/test_sync_* alone: 33 files, 726 tests, ~19.5k LOC). Largest: test_sync_apply.py 2,745 LOC/105 tests; test_slides_sync.py (CLI) 1,618 LOC/71 tests; test_sync_limitations.py 1,491 LOC/40 tests.
- Scenario expression: raw f-string builders of percent-format cell headers, redefined per file — 39 files define local `_slide`/`_write_pair`/`_deck` (53 occurrences); ~15 files each re-implement watermark seeding (`_seed_watermark`/`_seed`) with comments like 'Mirrors sync_apply._record_watermark exactly', in >=3 divergent schema generations; other files import the private production `_record_watermark` directly. No shared fixture/DSL library for sync scenarios (tests/slides/fixtures/ exists but sync tests barely use it). No hypothesis/property-based tests anywhere in the sync suite.
- White-box coupling: tests import >=15 private underscore symbols from the engine (_record_watermark, _resolve_alignment, _apply_ledger_overlay, _baseline_from_watermark, _shared_hashes, _conflict_as_edit, _maybe_retag, _plan_exit_code, _apply_exit_code, _outcome_line, _assign_item_ids, _eligible_for_partial_advance, _merge_notes, _require_slide_ids, _retag_direction), plus 23 monkeypatch/patch sites; refactoring engine internals breaks tests even when behavior is preserved.
- Regression accretion: 38 distinct issue numbers referenced 189 times across 30 sync test files (#216 x18, #438 x16, #190 x16, #365 x14, #289 x13, #364 x11, #228 x10, #269 x9 ...). Whole files named after incidents: test_sync_issue_269.py, test_sync_idless_localized_drift_repro.py (#364/#365/#366 stale-watermark incident); issue-named test functions test_issue443_*, test_issue454_* inside test_sync_limitations.py. Each is a hole the engine let through and later got a pinned fix.
- Combinatorial special-casing visible in test shape: test_sync_plan.py enumerates ~30 cold-start cases across baseline-source {watermark, git-head, none} x id-state {id-less, id-carrying, half-id'd, mismatched-ids, mixed} x provider {present, absent} x commit-state (TestBothDirectionsRefusal, TestReconcileEmission, TestBuildSyncPlanGitFallback). test_sync_issue_269.py needs 10 distinct cell-shape builders (_title/_ncode/_nmd/_nmd_tagged/_idless_code/_idless_md/_hdr/_sub_md/_sub_code/_idd_code) and runs every scenario under both baseline sources — the cell-class taxonomy is the engine's real interface.
- Invariants strongly pinned: (1) no-op => zero bytes + zero LLM calls (corpus backstop); (2) #269 propagate-or-alert with an explicit `_falsely_consistent(plan,result,propagated)` forbidden-state predicate reused across 4+ files; (3) apply-then-rerun idempotency (single round only); (4) dry-run/apply parity (#216) — preview must predict the writing run; (5) erroring pass writes NOTHING (buffered temp-swap, flushed=False, truthful 'rolled back' CLI line); (6) untouched cells stay byte-exact incl. terminal-newline edge cases; (7) both-sides-edited => deferred conflict, #365 degradation to per-cell conflicts, #447 policy tiers with equivalence gate and escalate containment; (8) verify = structural gate (id set-symmetry, duplicate ids, dropped-id warnings, exit 0/2); (9) ledger trust-overlay provenance (structural/agent/semantic oracle upgrade paths).
- Invariants MISSED: no multi-round convergence tests (no fixpoint property, no alternating-direction edit sequences over several syncs); no crash-injection between the two per-file os.replace calls (a crash leaving deck.de written but deck.en stale is untested); no fuzzing of deck shapes; separated-companion (voiceover_*.de/en.py) apply write-back untested in this tree (Phase 1 pins only that apply REFUSES — worktree HEAD 3980f9ab predates PR #515 Phase 2); `clm voiceover sync` companion tests bypass the pipeline by stubbing _merge_notes/merge_batch with AsyncMock.
- Oracle infrastructure (#298/#299 remediation): two real oracles — corpus no-op backstop (test_sync_corpus_noop.py: any no-op plan applies with zero bytes/zero LLM over 212 real pairs, plus regression floor noop_pairs>=40) and corpus mutation oracle (test_sync_corpus_mutation.py: 6 scripted one-sided EN mutations x <=3 clean pairs per cell-class, asserting propagate-or-alert + watermark-recorded + not-all-alerted). The mutation oracle caught two real failures (#443, repro'd at test_sync_limitations.py:1390-1491). BUT: real-corpus runs are `slow`-marked and never run in CI; the CI fallback corpus is a single 14-line deck pair (tests/data/sync_corpus/), on which all scale floors are skipped; mutations are EN-side only, never DE-side or both-sided.
- Channel-coverage meta-test (test_sync_tag_drift.py:396-542, the #289-class closer): a hand-maintained CHANNEL_COVERAGE dict mapping ~28 (partition,field) watermark channels to named private engine functions; a _RecordingCache asserts every recorded channel is registered and rows aren't widened silently. Verification strength is weak — `hasattr(module, name)` only proves the named function still exists, not that it consumes the channel; the map must be manually re-keyed on any engine refactor.
- Embedded corpus measurement contradicts engine health: Phase-0 (2026-06-01) found only 81 of 212 real PythonCourses pairs post-sync-clean — 131 pairs (62%) carry id-less narrative cells the engine cannot bring to a clean state; the regression floor (40) tolerates losing half of the working population (test_sync_corpus_noop.py:33-39).
- API smells surfaced by tests: conflict decisions keyed by Python object identity `{id(proposal): DECISION_DE_WINS}` (test_sync_conflict_policy.py:78) — plans are not serializable for decision round-trips (memory notes #447 apply_plan(conflict_decisions=...) overlay vs decisions= is itself a landmine); corpus tests hardcode the maintainer's personal path C:\Users\tc\...\PythonCourses\slides as a corpus fallback; compat tests for 'pre-198 watermark without tags' pin schema-generation back-compat in the cache.
- No xfail/skip markers encode known-broken behavior in tests/slides sync files (skips are environmental: git-on-PATH, corpus presence) — the suite is green-by-construction; known limitations live as prose in test_sync_limitations.py docstrings instead.
- Essential vs accidental split as evidenced by tests: essential = the cell-class taxonomy (neutral/localized x id'd/id-less x code/markdown x slide/voiceover/notes/header roles) and needing SOME baseline for 3-way merge; accidental = the out-of-band SQLite watermark baseline that can go stale (the entire #364/#365/#366 incident class and the rebaseline/--since/diagnose sub-suites exist only because of it), the 5-way baseline-source multiplicity, the id-state-driven refuse/reconcile/add matrix, the hand-maintained channel registry compensating for untyped watermark channels, and 15 divergent copies of watermark-seeding logic in tests.

## Pain points

- Baseline/watermark statefulness dominates the suite: ~15 per-file re-implementations of watermark seeding in 3+ schema generations, dedicated sub-suites for staleness (test_slides_sync_rebaseline.py, _since.py, _diagnose.py, test_sync_idless_localized_drift_repro.py), and every plan-level scenario duplicated across watermark vs git-HEAD baselines — a test-tax paid on every change because the baseline is out-of-band mutable state rather than derived from git.
- Cell referencing/alignment has no single abstraction, and tests prove it: identity is variously (slide_id, role) keys, positional pairing, content_hash, AST 'construct' slugs, narrative 'anchor' occurrence-under-slide, and id(proposal) — each with its own classifier path (_classify_narratives, _classify_idless_localized_drift, align_anchored, _moved_keys, _maybe_retag, ...) and its own test cluster; the CHANNEL_COVERAGE registry is a hand-built index over this sprawl.
- Combinatorial cold-start matrix (~30 enumerated refuse/reconcile/add cases in test_sync_plan.py) signals that direction-resolution is special-cased per id-state rather than derived from a uniform model; every new cell class or baseline source multiplies the matrix.
- Oracle strength is CI-hollow: the only tests that exercise real decks (no-op backstop + mutation oracle) run against a 1-deck synthetic fallback in CI; full-corpus validation happens only on the maintainer's machine at release time, and mutations are one-sided EN-only.
- Meta-test verification is name-based (hasattr) — the channel-coverage guard can pass while a detector function exists but no longer consumes the channel; it converts refactors into registry-editing chores without proving behavior.
- Massive white-box surface (>=15 private imports, 23 patch sites, CLI exit-code helpers imported directly) means any engine restructure — e.g. replacing the ad-hoc alignment with a systematic model — invalidates large parts of the 24k-LOC suite even where behavior is unchanged; the tests protect the implementation, not just the contract.
- Idempotency is tested only one step deep; convergence over realistic multi-session workflows (edit DE, sync, edit EN, sync, edit both, sync) is untested, which is exactly the agent-stuck-loop failure mode the maintainer reports.
- Multi-file write atomicity is tested only at the 'erroring pass buffers and writes nothing' level; there is no test for a crash between the per-file os.replace calls of a 2-file (soon 4-file) flush — the crash-safety of the pair/quad write is asserted by design, not by test.
- Voiceover-companion blindness (the recently discovered class of bugs) is now pinned for Phase-1 read modes, but the fix's tests live in yet another parallel representation layer (project_pair/Representation with MIXED/CROSS_LANGUAGE/SEPARATED refusals) — a 4th identity/representation regime layered on rather than unifying the existing three.
