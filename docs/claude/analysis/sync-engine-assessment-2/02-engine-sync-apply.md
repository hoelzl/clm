## Summary

Read all 4399 lines of src/clm/slides/sync_apply.py and all 412 lines of src/clm/slides/sync_plan_walker.py (worktree curious-wandering-crescent, master @ 3980f9ab = #501 Phase 1; the #515 companion write-back is NOT in this tree — apply_plan refuses any companion-aware plan outright at sync_apply.py:369-376).

apply_plan is a single ~350-line orchestrator with three whole-plan short-circuits (mint/adopt/reconcile, each with its own verify+write+watermark path, returning before FileState even loads), followed by a strictly-ordered main pipeline: snapshot content indexes -> load FileState for both decks -> materialize ALL model calls up front ("resolve-then-apply", #216) -> per-proposal dispatch (remove/edit/retag apply-or-defer, moves collected, conflicts resolved/deferred, refuse always deferred) -> adds (before moves, "sticky" via stamped ids) -> moves (only if _pass_is_clean) -> deterministic id-migration (§9) -> structural region rebuild (apply_code_structure in sync_code) -> group-order reconciliation (clean-pass-gated) -> four post-hoc fail-safe invariant checks that append errors -> atomic 2-file flush gated on zero errors -> full or partial watermark advance. The decision overlays differ sharply: `decisions=` (interactive walker) is a complete map where a MISSING entry defers even deterministic edits (_accepted, :663-671), while `conflict_decisions=` (#447 --conflict policy) is a conflict-only overlay consulted first in _conflict_decision (:696-713) with decisions left None so deterministic kinds keep the batch auto-apply default — this is exactly the recorded landmine (passing a policy via decisions= would defer every deterministic edit). Both maps are keyed by id(proposal), so they silently miss (conflicts silently defer) if built from any plan object other than the one passed to apply_plan (:765-767).

Idempotency is achieved by a constellation of per-mechanism guarantees rather than one model: full watermark re-record after a clean flush (re-parses the just-written files, 5 partitions + tags + anchors + synced_commit, :4121-4197); a per-cell partial advance for content-only passes that pins deferred keys at the old baseline hash and rewinds tag-holds (:4281-4399), guarded by a completeness invariant len(deferred_keys)==result.deferred (:647) that silently fails for ANY id-less deferral (since _note_deferred :657-660 records nothing for slide_id None), forcing a whole-watermark hold and full re-surfacing; add-stickiness via ids stamped onto both decks; and moves/group-reorders that only commit on a fully clean pass because "a reorder is idempotent only once the watermark advances" (:13-16, 3097-3110, 3169-3177). A deferred-but-error-free pass still writes (partial application is the designed outcome, :605-618); only an error rolls the whole 2-file write back — including errors raised by the fail-safes and by inexpressible moves, so one un-propagatable divergence discards all completed deterministic work for the pass.

The verdict evidence: the essential core (baseline diff, directed translation, conflict deferral, atomic write, watermark) is sound and heavily defended, but the file contains at least five coexisting cell-identity/reference schemes each with its own resolver, applier, doubling-guard, and fail-safe, four hand-mirrored materialize/execute walk pairs whose only correctness argument is a prose "superset" claim, four distinct disk-write paths (one non-atomic), and roughly a dozen manually-maintained detector/apply mirror invariants — the concrete substance behind the maintainer's "cell referencing feels ad-hoc" complaint.

## Key findings

- APPLY PIPELINE ORDER (sync_apply.py apply_plan :300-654): (1) companion_aware refusal :369-376 — separated-voiceover decks error out, nothing written, in THIS tree; (2) whole-plan short-circuits mint :383 / adopt :391 / reconcile :402 return before FileState.load; (3) content_index snapshots :408-409 + FileState.load :411-412; (4) _materialize_edits :419 resolves every edit/decided-conflict model call up front keyed by id(proposal); (5) narrative_targets pre-mutation snapshot :445-448 (object-identity mutation, #403 Phase B); (6) dispatch loop :450-510; (7) scope_to_proposals early flush-and-return :512-528; (8) _apply_adds :541 (before moves so new slides join reorders); (9) _apply_moves :546 (last, clean-gated); (10) _migrate_drifted_ids :558 BEFORE structural pass so corrected ids ride to twins; (11) _materialize_structural :566 + apply_code_structure :567; (12) _reconcile_group_order :585 (clean-gated); (13) fail-safes :590-601; (14) error-gated atomic flush :616-618; (15) watermark :637-652.
- DECISION OVERLAYS DIFFER BY DESIGN: `decisions=` is the interactive walker's COMPLETE map — when non-None, a deterministic proposal (edit/remove/retag/move) applies ONLY on explicit DECISION_APPLY (_accepted :663-671), everything unlisted defers. `conflict_decisions=` (#447) is a conflict-only OVERLAY consulted FIRST in _conflict_decision :696-713 while decisions stays None, so deterministic kinds keep batch auto-apply and only conflicts are steered. conflict_policy_decisions :751-778 deliberately omits id-less-localized and remove-vs-edit conflicts (they always defer). Both maps keyed by id(proposal): must be built from the exact plan object passed to apply_plan or every key silently misses and conflicts silently defer (:765-767 docstring warning).
- '_pass_is_clean' :889-896 = has_baseline AND no plan errors AND no result errors AND deferred==0. It gates three things so they never drift apart: _apply_moves :3110 (deferred moves otherwise), _reconcile_group_order :3177 (protects an interactively-skipped reorder from being silently re-applied — docstring :3169-3175), and the full watermark advance :638. Additional ordering invariants: adds-before-moves (:536-541), id-migration-before-structural (:554-557), materialize-before-mutate (stage 2b at :419, :2398, :2424-2425, :566), narrative-keys snapshotted before mutation (:445, :2749).
- IDEMPOTENCY MECHANISMS: full watermark advance re-parses flushed files and records 5 partitions (de/en/shared/de-header/en-header) + tag maps + narrative anchor maps + synced git commit (_record_watermark :4121-4197). Partial advance (_record_watermark_partial :4281-4399) pins deferred (slide_id,role) at old baseline hash and rewinds #202 tag-holds; eligibility (_eligible_for_partial_advance :4214-4254) requires content-only plan, no blocking_issues, real baseline, and something-to-bank. Completeness invariant :647 (len(deferred_keys)==result.deferred) FAILS for any id-less deferral because _note_deferred :657-660 skips slide_id None — so an id-less localized conflict or refused proposal forces a whole-watermark hold and everything (incl. reconciled edits) re-surfaces + re-invokes the judge next run: a real convergence weakness, safe but non-convergent.
- CONVERGENCE DEPENDS ON THE JUDGE AS EQUIVALENCE ORACLE: in_sync verdicts advance the watermark for conflicts (_conflict_already_equivalent :1689-1723 probes BOTH directions; _idless_conflict_already_equivalent :1726-1757); a non-deterministic judge flips cells between in_sync (banked) and deferred (re-surfacing) across runs. ## Pain points

- ACCIDENTAL: five cell-identity schemes with five parallel resolver/applier/guard stacks in one file. The task's core question ('is cell referencing ad-hoc?') is answered yes by structure: (slide_id,role) keys, narrative occurrence-under-slide keys, positional ordinals for id-less localized cells, construct-name anchors, and content-hash anchors all coexist, and every new cell shape (#198 retag, #365 id-less, #403 narratives) added another positional identity plus its own doubling guard. A single stable per-cell identity (universal ids, or one anchor algebra) would collapse most of _apply_retag_idless/_resolve_idless_localized_edit/_narrative_keys_by_index/narrative_cells_by_key/_migration_target and their fail-safes.
- ACCIDENTAL: id(proposal)-keyed decision maps across module boundaries. Python object identity as the API key means any re-plan invalidates every decision silently (conflicts just defer, no error) — documented as a landmine in conflict_policy_decisions :765-767 and encoded in project memory (#447). A value-keyed proposal identity (slide_id/role/kind or a plan-issued ordinal) would make the failure impossible.
- ACCIDENTAL: two overlay parameters (decisions= vs conflict_decisions=) whose semantics for an ABSENT entry are opposite (defer-everything vs batch-auto-apply). This is exactly the kind of surface that gets agents stuck; it exists only because the walker's complete-map contract was retrofitted with a non-interactive conflict policy instead of unifying on one decision structure with an explicit default policy field.
- ACCIDENTAL: four materialize/execute walk pairs whose correctness is a prose 'superset' claim plus an inline fallback 'not expected to fire' (:2536, :2867). The resolve-then-apply split (good idea: no model calls after mutation begins) was implemented by duplicating each execute walk's selection logic instead of deriving both from one enumeration, so every walk change must be made twice.
- ACCIDENTAL: four separate disk-write paths with different atomicity (_flush_states_atomically temp-swap; reconcile's own flush; adopt's plain FileState.flush write_text at sync_writeback.py:805-809 — non-atomic; mint via assign_ids_in_split_pair). The whole-plan short-circuits also duplicate the flush+watermark epilogue with subtly different gates (reconcile advances on its own deferred==0; mint/adopt advance immediately).
- ESSENTIAL-BUT-OVERGROWN: the watermark. A baseline is essential for 3-way sync, but this one is 5 partitions x (rows, tags, anchors) + synced_commit in an external cache DB, plus a per-cell partial-advance mode with a completeness invariant (len(deferred_keys)==result.deferred :647) that silently disables itself for any id-less deferral — safe, but it makes convergence behavior depend on which identity scheme a deferred cell happens to use. The planned git-committed ledger (#448) would subsume most of this.
- ACCIDENTAL: whole-pass rollback on fail-safe errors. The post-hoc parity checks (:590-601) and inexpressible-move errors (:3117, :3137-3141) append to result.errors, which un-writes ALL completed work including deterministic edits. Correct-by-paranoia, but it converts 'one cell the engine cannot express' into 'the whole apply did nothing', which is a primary agents-fall-back-to-manual-fixes driver. A per-cell commit unit (or writing the clean subset and reporting the rest) needs the identity/ledger rework first.
- ACCIDENTAL: ~12 documented mirror invariants (detector mirrors applier, agent surface mirrors verifier feed, three shared-cell partition functions must stay identical) maintained by comments rather than shared derivations or types. Several already share helpers (_build_slide_pairs, _migration_target, _reconcile_cross_product — the right pattern), but the remainder are drift hazards that historically produced exactly the class of bugs the issue log records.
- ESSENTIAL: the conflict taxonomy itself (both-edited, remove-vs-edit, id-less both-sided, tag-only, equivalence-downgrade, escalate) is inherent to bilingual co-edited decks; but its dispersal across _materialize_edits/_apply_conflict/conflict_policy_decisions/_is_* predicates (6 call sites consulting overlapping predicates) is accidental.
- OBSERVATION for the salvage question: the fail-safe layer, atomicity, resolve-then-apply staging, and safe-abort discipline are high quality and worth keeping; the unsalvageable-as-is part is the identity layer — everything downstream (partial watermark, doubling guards, positional retag/edit paths, group-order reconcile, the id-migration tiers) is compensating machinery for cells that lack stable identities. sync_plan_walker.py by contrast is trivially fine.
- SCOPE NOTE: in this tree the engine cannot write voiceover companions at all (companion_aware refusal :369-376) — 'many commands ignored voiceover sidecars' is structural here: the 4-file problem is handled as a 2-file engine plus a projection, and apply refuses the projected case.
