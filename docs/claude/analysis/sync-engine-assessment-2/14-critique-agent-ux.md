> Evidence slice of the 2026-07-01 sync-engine architecture assessment

> the current tree before reuse (tree at review time: `3980f9ab`, pre-#515).

## Position

Manual Claude-Code syncing beats the tool because the agent's native contract is "pure functions over git plus explicit per-cell decisions," while the sync surface is a stateful multi-invocation session over three invisible trust stores with fragile positional item handles and all-or-nothing writes — the byte-level write/verify core is worth keeping, but the surface and identity layer must be replaced by a 3-4-verb, value-keyed, git-baselined contract in which the tool owns only what the agent is bad at.

## Direct answers

### Is the current design salvageable?

Conditionally yes — by keeping the bottom and deleting the middle. Worth keeping as-is: the byte-faithful FileState write primitives and atomic temp-swap flush (sync_apply.py:3271-3309, verified flush gate at :616-618), the verify structural gate (id symmetry, duplicate ids, shared-cell byte parity — the one verb dogfooding says held), the split/unify round-trip contract (the only component with a declared algebraic invariant, Hypothesis-pinned), the hard validation gates on LLM output (validate_alignment's 6 gates, safe-abort discipline), and the translation prompts. Not salvageable by patching: the identity layer (7-8 coexisting schemes selected by incidental metadata) and the stateful surface (path-keyed watermark + its six repair mechanisms, the task/accept round-trip keyed by re-minted positional ids, autopilot as a parallel 26-option engine). The evidence pattern is decisive: ~80% of ~35 defects in 4 weeks are identity/state/scope-model bugs, not coding errors, and every fix added another hand-enumerated detector — the enumeration process itself is the defect generator. Salvage = keep the write/verify core, rebuild the surface as 3-4 stateless verbs, and unify identity; do NOT attempt to incrementally harden the current 9.2k-line plan/apply pair into correctness.

### Are the abstractions correct?

No, at two independent levels. (1) Cell identity: at least 7 coexisting schemes ((slide_id,role) keys, occurrence-under-slide ordinals, positional ordinals, content-hash anchors, construct slugs, vo_anchor tokens, id(proposal) object identity), selected per cell by incidental metadata (does it happen to carry an id? a lang attr? is it j2?), computed redundantly on plan/apply/baseline sides and held in agreement by 'must mirror exactly' docstrings — with silent-drop bugs already shipped when they drifted (sync_plan.py:1416-1419). The two biggest recent bugs (#443 cross-half id-asymmetry, #501 companion files) fell outside the model's own axes, meaning the failure mode has moved from 'uncovered cell' to 'unrepresented axis'. (2) The surface abstraction: it models sync as a stateful session over hidden stores, when the agent's native model is pure functions over git plus explicit decision documents. A systematic model would have: total per-cell identity (canonical form REQUIRES stable ids on sync-relevant cells, minted by a dedicated tool verb — the ~13k-stamp objection is a one-time normalize commit, and the split format staying makes this tractable; deterministic derived identity only for the residue); ONE unified cell stream covering all 4 files via a projection that round-trips by construction (as split/unify already does — extract/inline must be rebuilt to that standard, not compensated with representation markers); ONE 3-way diff over that stream producing a plan derived once (no post-passes retracting proposals by object id); value-keyed items that survive replanning; and git as the only baseline. That model dissolves the majority of the file: the 10 parallel channels, the 4 position spaces, the plan-patch passes, and the watermark repair industry are all consequences of partial identity plus hidden state.

## Argument

WHY MANUAL SYNC WINS (verified against the code). When Claude Code syncs a deck manually, its entire state model is git: `git diff HEAD -- deck.de.py deck.en.py` shows what changed on each side, the agent reads both halves, judges direction and translation equivalence per cell — the exact job the engine outsources to an LLM judge anyway (report hardcodes `provider_available = True` with the comment "the agent is the verifier", sync.py:1818-1824) — writes the twin edits, and commits. Every piece of state is visible, diffable, recoverable; failure is local. The tool loses to that on four verified fronts.

(1) COMPREHENSION TRAPS (CLI surface). (a) Invisible state: the watermark lives in project-local SQLite keyed by absolute resolved paths — silently missed from every git worktree (#435), invisible from a topic-dir cwd (#477) — and staleness is indistinguishable from drift: `explain` re-runs the entire classifier against git HEAD just to make staleness visible (sync_plan.py:4796-4829). An agent cannot cat, diff, or reason about it. Three overlapping trust stores (watermark / git-HEAD / ledger) have opposite staleness semantics. (b) Options that flip semantics mid-loop — VERIFIED: report/task/accept default to git-HEAD with `--use-watermark` as an opt-in flag (sync.py:1817, 2595, 2648-2653, 2789), while `apply` has `--use-watermark/--no-watermark` defaulting ON (sync.py:2084-2088). Consecutive steps of the canonical loop classify the same deck against different baselines; the same flag name has different polarity per verb. Same trap inside the API: `decisions=` defers anything unlisted while `conflict_decisions=` auto-applies anything unlisted (sync_apply.py:663-671 vs :696-713) — the recorded #447 landmine. provider_available is True in report but False in apply (sync.py:2310-2318), so a cold pair is a "task candidate" in one verb and "refuse" residue in the next. (c) Outputs that don't say what to do next: bare "1 deferred" hiding crossed decks (#231), historically empty conflict excerpts forcing an agent to hand-extract 70 cells (#451, dogfooding), "0 changes" while companions diverged (#501), remedy text that was wrong (#364, #438), and the structural pass silently no-oping when direction is ambiguous (sync_code.py:126-127) — "sync did nothing and didn't say why". The docs themselves concede "most early-baseline conflicts are false — a consistent bilingual edit" (commands.md:2311, verified) — a ~96% false-positive rate in the one production dogfood (56 conflicts + 17 edits flagged, 3 genuine). Exit code 2 means four different things across verbs. (d) Failure modes with no recovery verb — VERIFIED: `accept` re-matches items by a 6-tuple of (kind, role, slide_id, direction, source_position, target_position) and raises "the deck changed since `report`; re-run it" on any drift (sync_accept.py:148-176); since positions shift after every write, a multi-item session is O(N) full re-plans with designed-in stale-handle rejections. And apply is all-or-nothing — VERIFIED: the flush gate skips the ENTIRE 2-file write on any error (sync_apply.py:616-618), and errors include inexpressible moves ("some moves are not expressible by a slide-group reorder — deferred", :3137-3141) and mixed-direction moves (:3114-3117). One cell the engine cannot express converts to "the whole apply did nothing" — the single strongest driver of fall-back-to-manual, because manual editing never loses completed work. The partial-watermark completeness invariant (len(deferred_keys)==result.deferred, :647) silently disables itself for any id-less deferral, so a deck with one stubborn id-less conflict re-surfaces EVERYTHING and re-pays the (uncached, sync_translate.py:166-169) translation cost every run — a non-convergent loop. The 406-line sync-agents.md orientation topic (verified) is itself the tell: a tool needing 406 lines of mental-model transfer before first use has externalized its complexity onto the agent.

(2) DIVISION OF LABOR — the tool currently owns the wrong half. The agent is demonstrably good at: reading git diffs (its native format), judging direction and translation equivalence (the engine already delegates this — the whole tier-3 design assumes it), writing translated prose/code comments, and understanding renames and moves semantically (no heuristics needed — the mtime tiebreak at sync_plan.py:1380-1400 and slide_id-set rename matching at :1963-2001 are guesses at things an agent answers by reading). The agent is bad at: byte-exact 4-file consistency (separator blanks, terminal newlines — the engine's byte-faithfulness machinery is genuinely valuable), id minting/bookkeeping across twins (#162: independent minting diverges), atomic multi-file writes, occurrence counting, and exhaustive structural checks (id symmetry, duplicate ids, shared-cell byte parity). The current tool inverts this: autopilot embeds four model clients to do the judging the agent does better, while the mechanical layer the tool should do perfectly refuses whole passes, refuses companion decks outright in this tree (sync_apply.py:369-376), and re-litigates decided cells. Manual sync works precisely because the agent keeps the judging and loses only the mechanical guarantees — which it approximates well enough that a tool offering broken mechanics plus opaque state is strictly worse.

(3) THE ENGINE'S GOOD PARTS confirm salvageability of the bottom layer: the atomic temp-swap flush, verify's structural gate (which dogfooding says held: 52/52 clean at the end), the accept-path safety engineering (multi-cell smuggling rejection, strict_single translator, sync_accept.py:185-247 — verified in the same read), the validated LLM tiers (6-gate validate_alignment), and split/unify's declared byte-identical round-trip. These are exactly the "own the mechanical part perfectly" components. What sits between them — 9.2k lines of plan/apply with ~8 identity schemes, 10 content channels, 6 plan-patching passes, and a watermark with six repair verbs — is the accretion the issue history attributes ~80% of defects to (11 identity-model + 7 unmodeled-scope + 10 state-management out of ~35, vs 4 ordinary bugs).

## Concrete plan

THE MINIMAL AGENT CONTRACT — the tool does exactly 4 things perfectly, everything else belongs to the agent:
1. `sync report DECK [--since REF|--baseline REF]` — a PURE FUNCTION of (working tree, git ref). Shows aligned pairs and per-side changes since the ref, with FULL excerpts on every item (both current cells for conflicts), ONE addressing scheme, and STABLE VALUE KEYS per item (slide_id/role/content-hash based — never list positions, never id(proposal)). No hidden store consulted; same input, same output, every verb, MCP included. The ledger, if kept, is a suppression overlay consulted identically everywhere.
2. `sync apply DECK --decisions FILE` — "here are my per-cell decisions (JSON: item key → {apply | skip | de-wins | en-wins | body: ...}), execute them mechanically": twin writes, id stamping, byte-faithful rendering, atomic ≤4-file batch write. PER-ITEM isolation: items that succeed are written, items that fail are reported individually with a reason — never roll back item A because item B is inexpressible. This single change (plus stable keys) removes the two verified fall-back-to-manual triggers.
3. `sync verify DECK` — the existing structural gate, kept nearly as-is: id set-symmetry, duplicate ids, shared-cell byte parity, companion linkage. Read-only, fast, machine-readable, the post-apply truth check.
4. `sync ids DECK` — deterministic identity ownership: mint/repair shared slide_ids on both halves (absorbing assign-ids/reconcile-vo-ids/mint/adopt), so identity bookkeeping is a dedicated verb, not an emergent side effect of apply.
The task/accept round-trip disappears: report's excerpts ARE the task framing, and the decisions file IS the accept payload — zero stateful invocations between read and write, no 6-tuple re-matching, no HEAD race across three processes. Autopilot becomes a documented ~30-line loop (report → agent judges → apply → verify), not a second engine entry with 26 options.
THE HUMAN CONTRACT: one command — `clm slides sync DECK` — that prints a one-screen summary of what it will do, applies everything mechanical, walks each GENUINE conflict with both texts side-by-side and a default, writes atomically, and ends with a truthful one-line outcome plus a clean verify. For that to be trustworthy the engine must have: (a) no hidden state that can rot (git ref + optional committed ledger only — delete the SQLite watermark and its repair industry); (b) propagate-or-alert BY CONSTRUCTION: one total per-cell identity and one generic 3-way diff over one unified cell stream covering all 4 files, so a new cell class cannot ship uncovered; (c) per-item failure isolation; (d) companions first-class via a projection that round-trips byte-identically (rebuild extract/inline to split/unify's standard).
JUDGED AGAINST THIS CONTRACT, the current loop scores: report ~70% (right idea — tiered, model-free, excerpted — but nondeterministic across verbs via baseline divergence and inconsistent ledger participation, and items are not stable handles); task/accept ~20% (three stateful invocations where the contract needs zero; convention-only coupling "flags must match the report that framed the item"); apply ~50% (superb byte mechanics, wrong isolation unit, refuses the 4-file case in this tree); verify ~90% (keep); autopilot ~10% (relocated legacy, not a composition); MCP ~15% (read-only, diverges from CLI defaults).
PHASES: Phase 0, ergonomics triage, no engine rewrite (~1-2 wks): unify baseline defaults across all verbs (git-HEAD everywhere, one flag polarity), serialize value keys into report items and match accept by value not 6-tuple, make ledger participation uniform, delete no-op flags, fix MCP report to match CLI. Phase 1 (~2-4 wks): collapse task/accept into report-excerpts + decisions-file apply; implement per-item apply isolation (write clean subset, per-item outcomes); demote autopilot to a script over the verbs. Phase 2 (~4-8 wks): identity unification — canonical form requires ids on sync-relevant cells (one-time normalize commit across the corpus), rebuild extract/inline round-trip by construction, delete the watermark (git + ledger only), keeping verify and FileState untouched. Phase 3 (opportunistic): re-derive the plan as a single model over the unified stream, delete the 6 patch passes and the mirrored materialize/execute walks; rebuild the test suite around the contract (behavior-level, corpus oracles promoted into CI via the committed synthetic corpus). KEEP: FileState/atomic writes, verify, split/unify, validation gates, translation prompts, ledger concept. DELETE: watermark + 6 repair mechanisms, task/accept verbs, autopilot-as-engine, plan-patch passes, per-file test watermark seeders, suggest-sync/watermark alias groups.

## Risks

- Per-item apply isolation weakens the whole-pass parity fail-safes (#269 cardinal invariant): writing a clean subset while a shared-cell divergence stands risks baking divergence into the baseline. Mitigation: redefine the fail-safes per-item (an item only commits if ITS post-state passes parity for the cells it touches) and keep verify as the mandatory post-apply gate — but this is real design work, not a flag flip.
- Stable value keys require stable identity first: Phase 0 keys built from slide_id+role+content-hash are only stable for id'd cells; id-less items still shift. Full stability arrives only with Phase 2 (ids-on-sync-relevant-cells), so Phase 0 must honestly mark which item keys are durable.
- The ids-everywhere decision was explicitly priced out (~13k-stamp churn, #190 §3; reopened trigger = 'verifier pain', which has now occurred). The one-time normalize commit pollutes git blame and forces a coordinated corpus + CLM release; downstream repos (PythonCourses, CppCourses, ...) must migrate in lockstep.
- Deleting the watermark loses its lead-HEAD property (sync-without-commit is remembered). Mitigation: the workflow already assumes agent commits after sync, and the committed ledger covers per-slide trust — but human users who sync repeatedly without committing will see re-reports until they commit.
- The ~24k-LOC white-box test suite imports 15+ private symbols and re-implements watermark seeding in ~15 files; Phases 2-3 invalidate large parts of it even where behavior is preserved. Contract-level tests (corpus no-op, mutation oracle, dry-run parity) must be ported FIRST or the refactor flies blind.
- Adoption risk: the bar is a workflow that already works (manual Claude Code). If Phase 0 doesn't visibly cut round-trips (no more re-run-report loops, no whole-pass rollbacks) within one release, agents' learned avoidance persists regardless of later architectural quality.
- TOCTOU windows remain in any design that reads, plans, and writes in separate processes; the decisions-file contract narrows but does not eliminate the race (mitigate by having apply re-verify each item's source hash before writing — cheap with value keys).
- Effort estimates are optimistic if the maintainer continues daily dogfooding on the same branch: the 4-week history shows ~1 new sync issue per day; a feature freeze on the sync surface during Phases 1-2 is probably necessary.
