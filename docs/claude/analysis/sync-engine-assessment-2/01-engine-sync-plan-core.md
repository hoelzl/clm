## Summary

sync_plan.py (4829 lines, 122 functions) is the plan/classify half of the sync engine: it resolves a baseline for a DE/EN split pair, diffs both halves against it, and emits a typed SyncPlan of Proposals (11 kinds) and PlanIssues. The core model is a 3-way diff keyed on (slide_id, role), but that key covers only one of roughly TEN content channels; every other channel (id-less narratives, id-less localized cells, language-neutral shared cells, j2 headers, and three separate tag channels) has its own hand-built drift detector with its own identity scheme, its own conflict semantics, and its own refusal/alert vocabulary. build_sync_plan is a ~420-line orchestrator that runs the keyed classifier and then ~10 ordered post-passes, several of which mutate the already-emitted proposal list by object identity (drop/replace by id(p)) — the plan is not derived once from a model, it is emitted and then patched.

Identity is the central weakness: at least 8 distinct identity/anchor schemes coexist (slide_id keys, 4 synthetic membership roles, occurrence-under-slide narrative keys, vo_anchor predecessor tokens demoted from identity to placement-only, content-hash-with-uniqueness-guard anchors, AST construct slugs, baseline-position join keys, and at least 4 incompatible position spaces that must be manually re-indexed to line up). The file carries sediment from ~27 distinct issue numbers (referenced ~200 times in comments); many functions exist solely to compensate for a prior silent-data-loss bug in another function (e.g. #443 alert exists because the #10 reconcile unconditionally cancelled adds; #285 under-move retag exists because Tier C retag is positionally unsound under the #282 move conflict; #438 short-circuit exists because #225's cold-start demotion made clean committed decks perpetually re-surface mint candidates).

The essential complexity is real: cross-language content can never hash-compare, so a 3-way baseline is mandatory; cells legitimately lack ids; neutral cells are shared byte-identical; tags are invisible to body hashes; baselines can be absent, stale, or renamed away. But the accidental complexity dominates: identity is optional and heterogeneous per cell rather than total, so every channel needs bespoke pairing plus bespoke guards against every other channel's operations (move × retag, move × neutral-edit, reorder × id-less drift, id'd × id-less narrative asymmetry). A model in which every cell had a single total identity and one generic 3-way diff ran over one unified cell stream would eliminate the majority of this file. The classifier also imports its core identity predicates (role_of, hash_cell, construct_of) from sync_writeback — the apply-side module — an inverted layering that confirms identity has no owning abstraction.

## Key findings

- DATA MODEL (10 types): BaselineCell (frozen, sync_plan.py:118) and CurrentCell (:138) — position, slide_id, role, content_hash, plus late-bolted fields tags/anchor/owning_slide_id/raw_index each tagged with the issue that added it. Proposal (:157) has 11 kinds (add/edit/retag/move/remove/conflict/rename/refuse/mint/adopt/reconcile), 3 dispositions (apply/refuse/pending), 4 different position fields (source/target/old/new), plus content_hash (rename only), tags (id-less retag only), conflict_subtype (#447, one conflict flavor only), anchor/owning_slide_id/anchor_occ (narratives only) — a union type whose fields are mostly None per kind. TagHold (:245) has two mutually-exclusive identity schemes (slide_id+role XOR position). PlanIssue (:265) similarly carries positional-localization fields set on exactly one issue subtype. BaselineBundle (:295), SyncPlan (:323), AnchorAlignment (:670), _WatermarkRow 5-tuple (:292), _NarrKey (:2568).
- IDENTITY SCHEMES (>=8 coexisting): (1) (slide_id, role) key for id'd cells (:2009); (2) 4 synthetic membership roles NEUTRAL/LOCALIZED x CODE/MARKDOWN for cells role_of() disowns (:89-95); (3) narrative identity = (owning_slide_id, role, occurrence-under-slide) (:2558-2568) — explicitly NOT the predecessor anchor, which was found unstable and demoted to placement/reorder-detection only; (4) vo_anchor predecessor token, retained as a channel and read at :2686-2688 where the result is discarded (`_ = (de_reordered, en_reordered)`) purely so 'the channel stays consumed' — dead-value code to satisfy the #289 recorded-but-unread coverage gate; (5) content-hash anchor with uniqueness Counter guards, used in >=3 places (:3509-3527, :3606-3646, :653 docstring); (6) AST construct slug construct_of for id-migration candidates (:4689-4714); (7) baseline-position as cross-half join key, valid only when no move exists (:3561-3592); (8) row_anchor id:/construct:/hash: strings for explain (:4662).
- POSITION SPACES (>=4 incompatible, manually re-indexed): watermark per-partition position counts ALL non-j2 cells; classifier position counts only real-role cells — _baseline_from_watermark (:1431-1479) must drop membership rows and re-enumerate because _resolve_duplicates compares positions by ABSOLUTE difference (:2119). Separate spaces: non-j2-lang index for id-less localized targeting (:901-921), raw_index into the full RawCell stream for apply-locate (:154), group_index (:872-891), id-less-cell index reused as anchor_occ (:1059). Functions must walk 'in lock-step (identical predicate/order)' with each other (:836, :911) — alignment by convention, not by construction.
- PIPELINE: build_sync_plan (:3912-4331) stages: (1) companion projection in-memory inline, #501 (:3982); (2) parse + narrative identity maps over RawCell stream (:3988-3996); (3) baseline resolution with 5 sources (watermark / git-head / git:<ref> / git-from:<ref> / none) + representation-match check (:3894) + 2 rename-recovery heuristics (uncommitted rename matched by exact slide_id-set equality against git-deleted files :1963-2001; committed rename followed to HEAD^ :1935-1949) + cold-start demotion _pair_is_unbootstrapped #225 (:3849) + #438 consistent-committed short-circuit (:4130-4143); (4) classify_changes keyed 3-way diff; (5) ~10 ordered post-passes: align_anchored neutral drift (:4212), idless-localized drift (:4252), header drift (:4262), move-target-edit conflict #282 (:4271), neutral tag drift #289 (:4281), Tier-C idless retags or the under-move variant (:4290), cold mint/adopt upgrades (:4316-4318), ledger overlay (:4328). Pass order is load-bearing: passes gate on plan.has_errors, on move counts, and on the direction earlier passes 'established'.
- 3-WAY COMPARISON: per-side state machine _state() -> same/edited/added/removed/absent (:2180-2188); the keyed loop (:2318-2443) enumerates side-state combinations: both-edited-hash-equal -> in_sync (#4), both-edited-differ -> conflict, one-edited -> directed edit, both-same -> retag check + LCS-complement move detection (:2191-2232), removed-vs-edited -> conflict subtype 'remove-vs-edit', removed-vs-same -> remove, present-vs-absent -> add, ambiguous added/edited mixes -> warning. Direction (de->en / en->de) = which side drifted from ITS OWN baseline; cross-language hashes never match so 'in sync' is asserted structurally, with translation-equivalence deferred to an apply-time LLM judge (:2333, :1051).
- PLAN-PATCHING PASSES (proposals emitted then retracted by id()): _alert_asymmetric_companion_drift drops adds (:2876-2893), _reconcile_narr_pairs cancels add pairs (:2954-2962), _replace_adds_with_refusals (:3022), _replace_adds_with_reconcile (:3051), _apply_ledger_overlay drops trusted edits/conflicts (:4376-4399), _maybe_emit_cold_mint/adopt remove refusals and append a single pending candidate (:4439-4451, :4572-4585). Six passes mutate plan.proposals after emission — direct evidence classification is not derived from a single model.

- SEPARATED-VOICEOVER (#501) is a projection bolted on top: project_pair inlines companions in memory before ANY classification (:3982), baselines must be projected identically at every git source (3 separate call sites pass project_companion :4034/:4041/:4064-4068/:4084), the watermark gains a representation marker whose mismatch silently demotes to git-HEAD (:3894-3909), mixed/cross-language pairs get a refusal-only plan (:3881), and the plan must carry projected_de_text/projected_en_text because every position in it indexes the projected stream, not the on-disk files (:359-364). ## Pain points

- Identity is optional and heterogeneous per cell — slide_id may be absent, and the two halves may DISAGREE on id-ness — so instead of one pairing algorithm there are ~8 identity schemes and every pairwise interaction (move x retag, reorder x neutral edit, id'd x id-less narrative, duplicate bodies x hash anchors) needs a bespoke guard. This is the root accidental complexity; ids-on-every-cell (or a total derived identity) would delete an estimated half of the file.
- Ten parallel content channels (keyed cells, id-less cells, id-less narratives, id'd narratives, neutral shared cells, id-less localized cells, j2 headers, 3 tag channels) each re-implement the same 3-way same/edited/added/removed logic with slightly different semantics. The keyed loop (:2318), _classify_narratives (:2646), and _classify_idless_localized_conflicts (:959) are three hand-written copies of the same state machine.
- Plan-then-patch architecture: 6 post-passes retract or replace already-emitted proposals by object id(). Correctness depends on pass ORDER and on each pass's knowledge of what earlier passes emitted (e.g. #443's alert must run before #10's reconcile 'so what reaches here is only the genuinely-unchanged pairs' :2913-2915). No single invariant defines what a valid plan is.
- Four incompatible position spaces aligned only by 'lock-step iteration with identical predicates' across separate functions (:836, :911, :3358) — a change to one predicate silently misaligns another. _baseline_from_watermark must re-index positions because ONE consumer compares absolute positions (:1437-1448).
- Watermark-vs-git baseline duality: BaselineBundle (#289 P1) was itself a repair after the two sources diverged in coverage three separate times (#269, #225/#226, #289 tag drop — documented at :295-312). Semantics STILL key on source (committed-pair logic gates on source=='git-head' :3166), so the unification is representational, not semantic.
- Heuristics standing in for identity: mtime tiebreak for divergence winner (:1396), exact slide_id-set match for rename recovery (:1979), min-absolute-position-distance for duplicate original selection (:2119), mass-add threshold of 3 (:2555). Each is a guess where the model has no answer.
- The vo_anchor channel is recorded, read, and its result discarded (:2688) purely to keep a coverage gate happy — a recorded-but-almost-unread channel kept alive after its identity role was revoked.
- Issue-number sediment: ~27 issues referenced ~200 times; most functions are named for the bug they fix rather than the concept they implement. Docstrings routinely explain the prior data-loss bug a function compensates for — the module is a changelog, and correctness rationale lives in prose that only holds while the described interactions stay frozen.
- Baseline staleness is not modeled: a stale watermark produces errors indistinguishable from real drift, worked around by re-running the full classifier against git HEAD inside the explain renderer (:4796-4829) and by prose advice in error messages ('the watermark is likely stale — apply auto-re-baselines it' :1150-1153).
- Voiceover companions have no first-class identity — sync sees them only through an in-memory inline projection (#501), so companion-awareness had to be threaded through every baseline source, the watermark schema (representation marker), and the plan itself (projected_*_text); the recent 'many commands ignored sidecars entirely' discovery is a direct consequence of the projection being opt-in per call site.
- Vestigial/asymmetric code: _classify_cold's `excluded` param is always passed set() (:2291); Proposal is a union-of-11-kinds where most fields are None for most kinds, so consumers must know per-kind field conventions documented only in the 60-line class docstring (:157-213).
