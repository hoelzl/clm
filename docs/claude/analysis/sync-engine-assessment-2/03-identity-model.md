## Summary

I mapped the identity/alignment model across the sync engine's core modules (pairing.py, anchor_primitives.py, sync_companion.py, sync_code.py, reconcile_vo_ids.py, sync_semantic.py, plus the identity primitives they import from sync_writeback.py, sync_plan.py, sync_apply.py, voiceover_tools.py). There is no single cell-identity scheme: at least seven coexist, selected per cell by incidental metadata (does it carry slide_id? a lang attribute? a narrative tag? is it j2?). (1) keyed `(slide_id, role)` for id'd cells (role_of, sync_writeback.py:55-89); (2) content anchors `id: > construct: > hash:` for id-less code (anchor_of, sync_writeback.py:296-314); (3) occurrence-under-slide `(owning_slide_id, role, occ)` for id-less narratives (sync_plan.py:2558-2601); (4) predecessor placement anchors `id:/fp:/tm:...#occ` (anchor_primitives.py) — explicitly placement-not-identity, but persisted into companion files as `vo_anchor`; (5) `for_slide` slide references in voiceover companions with a legacy fallback chain (voiceover_tools.py:1124-1137, 1344-1387); (6) positional/ordinal schemes (watermark per-partition positions, j2 ordinal matching sync_code.py:376-388, shared-cell positional hash diff sync_plan.py:690-762); (7) file-level `.de/.en` twin arithmetic in two parallel families (prefix-gated vs prefix-agnostic, pairing.py:128-274).

The handoffs between schemes are the fragile part. A narrative cell switches engines depending on whether it happens to carry a slide_id (id'd → keyed path; id-less → occurrence path), and reconcile_vo_ids.py exists solely to convert cells between those two regimes — with a THIRD occurrence-counting implementation that deliberately counts differently from the sync engine's two (over all narratives vs id-less only). Identity is recomputed independently on the live side, the baseline/watermark side, and the apply side, held in agreement by documented "must mirror exactly / lockstep / drift = silent misplacement" invariants rather than by shared code paths in several places (sync_plan.py:1403-1428 vs anchor_primitives.py:218-248; sync_apply.py:1379-1401 vs sync_plan.py:2574-2601; watermark_rows/watermark_tag_map/watermark_anchor_map iterating in parallel, sync_plan.py:544-635). Two distinct content-hash functions with different normalizations coexist (body_fingerprint vs hash_cell). Four position coordinate systems interconvert via re-indexing (stored watermark positions → legacy positions → CurrentCell.position → RawCell raw_index), with `_baseline_from_watermark` (sync_plan.py:1431-1479) documenting exactly why the spaces must match.

Verdict on the maintainer's question 2: this is accumulated heuristics with locally-principled patches, not a principled scheme. The keyed `(slide_id, role)` core is sound but covers only id'd cells; every other cell class got its own detector added reactively per issue (#166 keyed engine, #190 anchors + shared-hash direction, #198 tag map, #199 vo_anchor, #246/#247 title/j2 special cases, #269 idless multisets + signature checks, #403 occurrence keys + Phase-B snapshot bugfix, #429 reflow hashing, #501 companion projection). Essential complexity (bilingual 3-way merge with partial ids and 4 files) explains maybe a third of what's here; the rest is the cost of never unifying identity: the same "which cell is this?" question is answered by different algorithms in different modules for different cell classes, and the module sizes (sync_plan 4829 + sync_apply 4399 lines) reflect the fan-out. A systematic redesign around one canonical per-cell identity (stable key when present, deterministic derived key otherwise, computed once and carried through plan→apply→baseline) would eliminate the largest bug class the docstrings themselves keep warning about.

## Key findings

- IDENTITY SCHEME COUNT: >=7 distinct cell-identity/alignment schemes coexist, dispatched per cell by role_of (src/clm/slides/sync_writeback.py:55-89): (a) keyed (slide_id, role) for narrative-tagged/id'd markdown and id'd localized code (synthetic CODE_ROLE, line 52); (b) content anchor 'id:>construct:>hash:' for id-less code (anchor_of, sync_writeback.py:296-314, with a row-side twin row_anchor:317-328 that 'must stay in lockstep'); (c) occurrence-under-slide (owning_slide_id, role, occ) for id-less narratives (sync_plan.py:2558-2601); (d) predecessor placement anchors id:/fp:/tm:<val>#<occ> (anchor_primitives.py:36-171) — used for placement and reorder detection, explicitly NOT identity (sync_plan.py:2558-2567); (e) for_slide slide references in voiceover companions (voiceover_tools.py); (f) pure positional/ordinal matching (watermark partition positions sync_plan.py:544-635; j2 cells matched by ordinal sync_code.py:376-388; shared cells by positional hash-sequence diff align_anchored sync_plan.py:690-762); (g) file-level .de/.en twin derivation (pairing.py:128-274).
- A cell's identity REGIME depends on incidental metadata, not its kind: an id-carrying narrative syncs on the keyed (slide_id,role) path while an id-less one syncs on the occurrence path (sync_plan.py:2591-2593 'Only id-less narratives are anchor-managed'). reconcile_vo_ids.py exists solely to migrate narratives between the two regimes (strip/stamp slide_id), and collapse_intra_half_duplicates (reconcile_vo_ids.py:217-243) exists because over-stamping collides the keyed regime's (slide_id, role) key while the occurrence pairing 'sees them agree'.
- THREE separate occurrence-counting implementations: sync_plan._index_narratives_by_anchor (id-less only, skips duplicated slides; sync_plan.py:2574-2601), sync_apply._narrative_keys_by_index (id-less only; sync_apply.py:1379-1401, documented as 'the apply-side mirror'), and reconcile_vo_ids._narrative_index (counts over ALL narratives id-less and id'd alike — deliberately different semantics; reconcile_vo_ids.py:85-107). So 'occurrence-under-slide' is not one identity: it is two subtly different ordinal spaces depending on which tool computes it.
- TWO owning-slide derivations that must agree by discipline: anchor_primitives.owning_group (live cells, language-aware, title-macro fallback; anchor_primitives.py:218-248) vs sync_plan._baseline_owning_by_position (watermark row walk; sync_plan.py:1403-1428) whose docstring says 'This must mirror owning_group exactly, or ... a one-sided edit is silently dropped' and records a bug that already happened ('keying on the anchor alone left it None and unpaired').
- TWO content-hash functions with different normalizations: body_fingerprint (12-hex sha256, drops ALL blank lines, hashes j2 header as body; anchor_primitives.py:66-91) for placement anchors, vs hash_cell/cell_content_hash (full sha256, reflow-normalized markdown, comment_token-dependent, byte-exact code; sync_writeback.py:255-273) for watermark/drift. Changing either canonical form 'invalidates every stored hash — bump WATERMARK_HASH_VERSION' (sync_writeback.py:257-258) — identity is entangled with persisted-format versioning.
- FOUR position coordinate systems: stored watermark positions (all non-j2 cells per de/en/shared partition), 'legacy-only position space' (real-role cells only, re-indexed by enumerate in _baseline_from_watermark; sync_plan.py:1437-1446 documents that _resolve_duplicates compares positions by ABSOLUTE difference so the spaces must match), CurrentCell.position (per-lang role-bearing cells; sync_plan.py:497-541), and RawCell raw_index — plus the load-bearing unchecked invariant 'parse_cells and split_cells yield the same cell sequence' aligning Cell and RawCell streams position-for-position (sync_plan.py:483-484, 618-620).
- Watermark stores identity as THREE parallel position-keyed maps that iterate 'identically so the two never drift out of position lock-step': watermark_rows (5-tuples), watermark_tag_map (#198), watermark_anchor_map (#403) — sync_plan.py:544-635. Keys are positions, not identities; every consumer re-derives identity from the arrays.
- Occurrence ordinals are UNSTABLE UNDER MUTATION: 'the occurrence ordinal is a position ... so it shifts the moment one is deleted'; execute-time appliers must resolve targets from a pre-mutation snapshot by Python object identity (narrative_cells_by_key, sync_apply.py:1427-1439) — an already-shipped bug class (Issue #403 Phase B bugfix: 'silently mis-targets the second of several removes/edits under one slide').
- Voiceover companions reference slides via for_slide=slide_id plus optional vo_anchor, with a 4-step legacy fallback chain in _plan_insertion (voiceover_tools.py:1344-1387): vo_anchor match -> for_slide group-end -> title-intent fallback (which accepts BOTH for_slide="title" and the historical slide_id="title"-with-no-for_slide form, voiceover_tools.py:1124-1137) -> unmatched. Extraction REQUIRES slide_ids and mints them (_ensure_slide_ids), with the #162 hazard that independent minting on the two halves diverges — the reason reconcile_vo_ids 'never mints a fresh id' (reconcile_vo_ids.py:12-19).
- Sync sees companions only via an in-memory INLINE PROJECTION (sync_companion.py:156-208, issue #501): companion cells are folded into deck text so the plan engine re-derives narrative occurrence keys over the projection; for_slide-space identity is erased and re-created. Pre-#501 the engine was 'structurally blind' to companions (sync_companion.py:3-8) — the maintainer's 'many commands ignored voiceover sidecars' finding is confirmed in-code. Mixed and cross-language representations are refused outright, and any unresolvable for_slide blocks the whole pair (lines 27-29, 193-200). Known landmine (project memory): inline-then-extract is not byte-identical.
- Structural pass (sync_code.py) is a SEPARATE alignment engine over region signatures: per-cell tuples ('R',role)/('S',body)/('L',kind)/('J',) (_signature, sync_code.py:246-293) — deliberately lossy differently per cell class — plus TWO drift detectors for id-less localized cells (construct-anchor hash mismatch AND Counter multiset membership, _region_has_localized_drift sync_code.py:296-342) because a hash-anchored cell's edit changes its own anchor. Gated on a single global run direction with two inference paths (keyed proposals else plan.anchor_direction from positional shared-hash diff; sync_code.py:119-127, sync_plan.py:690-762); no unambiguous direction => the whole structural step silently no-ops (sync_code.py:126-127).
- File-level pairing has TWO parallel derivation families — prefix-gated (split_twin/split_twin_pair) vs prefix-agnostic (split_lang_tag/order_split_pair/derive_split_twin/derive_split_pair/derive_split_pair_from_stem) — with pairing.py:120-125 admitting the logic 'used to live, copied' in 3 modules ('the four-way drift the duplication invited'), and the voiceover_ filename prefix special-cased in 3 separate functions (pairing.py:228, 268, 299).
- Duplicate slide_ids break the keyed scheme's uniqueness assumption and are resolved by a group-level heuristic: identify 'the original' by baseline content hash, rename the copies, else hard error (sync_plan._resolve_duplicates:2068-2110). Downstream consumers restate the constraint independently (mobile Studio: editable only if (slide_id, role) UNIQUE per file).
- role_of's tag set _SYNC_ROLE_TAGS is duplicated into sync_writeback.py:42-45 ('Duplicated from clm.slides.sync_plan to keep this low-level write module free of an import cycle') even though tags.py declares itself 'the single source of truth for all recognized cell tags' — the sync-role taxonomy lives outside the canonical taxonomy module.
- Module scale as accretion evidence: sync_plan.py 4829 lines, sync_apply.py 4399, voiceover_tools.py 1723, 15+ sync_* modules totaling ~19k lines; nearly every identity mechanism is annotated with the issue that forced it (#166, #190, #198, #199, #216, #242, #246, #247, #269, #289, #403 A/B, #429, #447, #448, #501), i.e. each new silent-drop bug added a channel/detector instead of revising the identity model.
- sync_semantic.py is orthogonal to identity: an LLM translation-faithfulness judge over already-paired (de,en) bodies for the #448 ledger; it consumes pair identity but does not define it (keyed by whatever the ledger recorded).

## Pain points

- Per-cell identity regime is selected by incidental metadata (presence/absence of slide_id, lang, narrative tag, j2-ness) rather than assigned canonically — the same logical cell migrates between keyed, occurrence, anchor, and structural engines when an author adds/removes an id; a whole module (reconcile_vo_ids.py) plus a duplicate-collapse pass exist just to shepherd cells between regimes. ACCIDENTAL.
- Identity is computed redundantly on live/baseline/apply sides and across tools, kept consistent only by 'must mirror exactly' docstring contracts: owning_group vs _baseline_owning_by_position, _index_narratives_by_anchor vs _narrative_keys_by_index vs _narrative_index (which intentionally differs), anchor_of vs row_anchor, three lock-step watermark maps. Every one of these is a documented silent-misplacement or silent-drop hazard when they drift — and one such drift bug is documented as having already occurred (sync_plan.py:1416-1419). ACCIDENTAL: a single identity computed once and persisted would remove the class.
- Position/ordinal coupling: four position spaces with re-indexing gymnastics (_baseline_from_watermark), an unchecked Cell/RawCell stream-alignment invariant, occurrence ordinals that shift under mutation (fixed by snapshot-by-object-identity rather than by stable keys), and absolute-position comparison in _resolve_duplicates. Fragile-by-construction. Mostly ACCIDENTAL — a consequence of storing positions instead of identities in the watermark.
- Two content-hash functions with different normalization/truncation rules (body_fingerprint vs hash_cell) mean 'same content' is answered differently by the placement layer and the drift layer; hash canonical-form changes invalidate all persisted baselines (WATERMARK_HASH_VERSION). Partly ESSENTIAL (placement wants edit-tolerance, drift wants sensitivity) but the split is undocumented at the model level and easy to conflate.
- Single global propagation direction per run: the structural pass and neutral-cell propagation silently no-op when the direction is ambiguous (sync_code.py:119-127), and direction itself is inferred by two different mechanisms (keyed proposals vs positional shared-hash classification). For agents this reads as 'sync did nothing and didn't say why'. Partly ESSENTIAL (both-sides-edited is genuinely ambiguous), but the silent no-op and dual inference are ACCIDENTAL.
- The companion boundary is a representation seam, not a shared model: companions live in for_slide/vo_anchor space, sync works in deck space, and the #501 projection converts one into the other in memory per run — with total-refusal semantics (one bad for_slide blocks the whole 4-file pair) and a known non-byte-identical round-trip. Pre-#501 blindness was the direct cause of the maintainer's 'commands ignored voiceover sidecars' discovery. ESSENTIAL that companions must be joined; ACCIDENTAL that the join is a per-run text projection rather than a first-class 4-file model.
- Heuristic fallback chains accumulate legacy forms: title identity alone has TITLE_SLIDE_ID (no id on the macro cell), tm:title#0 placement tokens, for_slide="title", and the historical slide_id="title"-no-for_slide companion form, all still honored (pairing.py:37-47, anchor_primitives.py:60-63, voiceover_tools.py:1124-1137). Each fallback is individually justified; the sum is unexplainable to a newcomer/agent. ACCIDENTAL accretion.
- Special-case lattice per cell class in the structural signature and rebuild (j2 by ordinal, shared by body, localized by kind, per-cell-synced by role-only; plus separator/blank-line preservation rules with per-class exceptions, sync_code.py:246-293 and 561-594) — every rule cites the issue that forced it; the model was never re-derived from first principles after the exceptions piled up. ACCIDENTAL.
- Bug-prone hot spots to weight in the salvage decision: (a) any code path that recomputes occurrence over a mutated deck; (b) any new watermark channel that must iterate in lock-step with watermark_rows; (c) baseline-vs-live identity derivation for narratives (already produced silent-drop bugs); (d) duplicate slide_id resolution; (e) direction inference interplay between keyed and anchor paths; (f) companion projection refusals and round-trip fidelity.
- What is genuinely ESSENTIAL here: a cross-language stable key for paired cells; some deterministic scheme for id-less cells; a 3-way baseline to distinguish edit-direction from conflict; language-aware group bounds for interleaved decks; companion->slide references. The current code solves all of these, but each with 2-3 competing mechanisms — the salvage question is largely 'can the mechanisms be collapsed to one per problem', not 'is the problem solvable'.
