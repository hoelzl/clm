## Summary

This slice covers the sync engine's file-projection and write-back layer (sync_writeback.py, split.py), its two bounded-LLM escalation tiers (sync_recover.py), and translation of new cells (sync_translate.py), plus the companion-projection machinery those files hook into (sync_companion.py, voiceover_tools.py, the #501 Phase 2 write-back in commit 52cb8a3b which is one merge AHEAD of this worktree's HEAD 3980f9ab — at HEAD, apply still refuses any companion-involving pair outright at sync_apply.py:369).

Projection model: the 4 files are never reconciled as 4 files. build_sync_plan (sync_plan.py:3969-3987) classifies the pair's voiceover representation via project_pair (sync_companion.py:156) into PLAIN / SEPARATED / MIXED / CROSS_LANGUAGE; MIXED and CROSS_LANGUAGE refuse. For SEPARATED, each half's companion is inlined IN MEMORY via inline_pair_text (voiceover_tools.py:218) — companion cells are placed after their owning slide by for_slide/vo_anchor and stripped of those author-only attrs so they look hand-authored — reducing the problem to the pre-existing 2-file inline engine. The engine's per-file model is FileState (sync_writeback.py:440): a byte-faithful list of RawCells with mutation primitives keyed by (slide_id, role), plus positional-ordinal fallbacks for id-less localized cells and object-identity fallbacks for anchor-keyed narratives. Write-back: plain pairs flush via per-file temp+os.replace (_flush_states_atomically, sync_apply.py:3271); separated pairs (Phase 2) render each inlined half, run extract_pair_text (voiceover_tools.py:957) to split it back into voiceover-free deck + companion, dirty-diff against disk, and commit surviving files in one atomic_write_all batch (path_utils.py:323), with companion deletions and other-layout pruning done non-atomically afterwards.

The inline∘extract non-identity is real and structural: extract mints slide_ids (_ensure_slide_ids, voiceover_tools.py:689), stamps for_slide/vo_anchor onto voiceover headers (_build_voiceover_header, :514), and globally squashes blank-line runs (re.sub(r"\n{3,}"), :738), none of which inline reverses. The code compensates with four interlocking mechanisms: re-extract BOTH halves whenever either changed; record the watermark over inline(disk-after-write) via _inlined_from_disk rather than the in-memory render; stamp a "separated"/"plain" representation marker into the watermark and demote to a projected git-HEAD cold start on mismatch (sync_plan.py:3894-3909, 4044-4053); and an empty-plan short-circuit that leaves non-canonical but in-sync files untouched. This is locally sound (the recorded baseline equals exactly what the next run will read) but is a symptom: the "canonical form" is defined operationally by extract's implementation rather than by any declared normal form, so every future change to extract's canonicalization silently threatens watermark validity — the same class of landmine the code already documents for normalize_for_hash (sync_writeback.py:255-258, "changing either branch's canonical form invalidates every stored hash").

sync_recover.py is NOT free-form LLM repair: it is two opt-in, validated, cached, safe-abort tiers. (1) AlignmentRecoverer (--llm-recover, Opus): body-free re-identification of slide_ids in an ambiguous code region, invoked only after the deterministic id-migration applied nothing (sync_apply.py:3469); the returned id↔cell map passes 6 hard gates in validate_alignment (sync_recover.py:227-294) or the region is left untouched and deferred. (2) CorrespondenceVerifier (--verify-cold-pairs, Haiku): body-bearing "are these two slides translations of each other?" verdicts gating cold-start shared-id minting. Translation (sync_translate.py) is full translation, not stubs: three role-specific prompts (prose / code / title — the title prompt exists because the prose prompt's "every line is prefixed with '# '" framing made models refuse to translate bare macro-argument titles, :329-341); target cells are built by cloning the source header and swapping lang (build_twin_cell, sync_writeback.py:416). With no translator, adds/renames are deferred (model-free agent path) or error (autopilot path) (sync_apply.py:2368-2375). Notably, the sync path runs the translator UNCACHED (sync_translate.py:166-169), so a repeatedly-deferred pass re-spends on translation.

## Key findings

- Projection model: 4-file sync is solved by reduction — separated voiceover companions are inlined in-memory into each deck half (inline_pair_text) so the pre-existing 2-file plan/apply engine runs unchanged over the projection; the plan carries projected_de_text/projected_en_text and apply loads FileState from those texts, not disk (sync_plan.py:4167-4170; Phase 2 diff to sync_apply, commit 52cb8a3b).
- Representation classification is a hard gate: PLAIN (identity projection), SEPARATED (inline+reconcile), MIXED (inline voiceover AND companion in one half) and CROSS_LANGUAGE (halves disagree on representation) refuse with normalize hints; an unresolvable for_slide is a total-transform refusal — narration is never dropped (sync_companion.py:110-137, 193-200). Inline `notes` beside a voiceover companion is deliberately NOT mixed (post-#387 steady state, sync_companion.py:101-107).
- Write-back is ≤4-file batched: render each half, extract_pair_text back to voiceover-free deck + companion, dirty-diff vs disk, one atomic_write_all (temp-all-then-serial-os.replace, path_utils.py:323-351); deck on disk is voiceover-free before AND after so a crash can never leave a half-inlined deck. But companion unlink (emptied narration) and _prune_other_companions run AFTER the atomic batch, non-atomically (commit 52cb8a3b, _flush_companion_writeback).
- inline∘extract non-identity has three concrete sources: extract mints slide_ids onto id-less content cells (twin-aware, voiceover_tools.py:689 via _ensure_slide_ids:459), stamps for_slide/vo_anchor onto companion headers (_build_voiceover_header:514) which inline strips (:273-279), and squashes blank-line runs deck-wide (re.sub(r"\n{3,}") voiceover_tools.py:738).
- Compensation stack for the non-identity: (a) re-extract BOTH halves when either changed, else the two companions end up in different canonicalization states and re-report as drift; (b) watermark recorded over _inlined_from_disk (re-read + re-project the just-written files) because extract canonicalizes headers inline doesn't strip, so recording the in-memory render would drift by exactly that delta; (c) watermark carries a representation marker ('separated'/'plain'); marker mismatch demotes to a git-HEAD baseline that is itself companion-projected at the ref (_inline_companion_at_ref, sync_plan.py:1748-1760); (d) empty-plan short-circuit leaves in-sync files byte-untouched. All in commit 52cb8a3b + sync_plan.py:3894-3909, 4044-4053.
- Verdict on the compensation: locally sound (fixed-point argument — recorded baseline == inline(disk) == what the next run reads; dirty-diff prevents churn of already-canonical files) but a symptom, not a fix: the canonical form is emergent from extract's code, not declared; the codebase already carries an identical landmine warning for hash canonicalization ("changing either branch's canonical form invalidates every stored hash — bump WATERMARK_HASH_VERSION", sync_writeback.py:255-258).
- sync_recover.py = two bounded LLM tiers, both opt-in/validated/cached/safe-abort: AlignmentRecoverer (--llm-recover, Opus via OpenRouter) re-identifies slide_ids in ambiguous code regions AFTER deterministic _migrate_drifted_ids applies nothing (guard at sync_apply.py:3469); it sees only (slide_id, construct-slug, content_hash) triples — never source — and its map must pass 6 gates (total, ids-from-base, injective, byte-identical cells pinned bidirectionally, NEW-needs-construct, no worn id dropped) in validate_alignment (sync_recover.py:227-294) or the region is deferred untouched.
- CorrespondenceVerifier (sync_recover.py:483-718, Haiku default) is the cold-start tier: before minting a shared slide_id onto a never-id'd structurally-aligned pair, an LLM confirms per-pair (heading + snippet, body-bearing unlike the recoverer) that DE/EN are actually translations; any 'no', parse failure, or non-total verdict map safe-aborts to refuse. Both tiers cache on a fingerprint of the exact serialization the model sees + prompt_version that folds in the model name (sync_recover.py:145-155, 410-414, 531-540).
- Translation is full-fidelity, not stubs (Issue #166 decision): SlideTranslator protocol with 3 role prompts — prose (preserve '# ' prefixes/structure), code (translate only human-facing strings+comments, identifiers byte-identical), title (bare phrase; exists because the prose prompt caused hallucinated '# ' prefixes and untranslated titles, sync_translate.py:329-341). Target cell = source header cloned with lang= swapped (build_twin_cell sync_writeback.py:416, swap_lang :359) inserted at the source-order anchor; brand-new structural inserts use build_cell (:375) which mints headers in canonical attribute order.
- No-translator behavior: adds/renames deferred as residue for the agent task/accept surface (deterministic_only) or hard error on the autopilot/human path (sync_apply.py:2368-2375); per-cell translation failures defer that cell and hold the watermark. The embedded sync translator is UNCACHED (sync_translate.py:166-169 — only the `clm slides translate` verb wraps it in CachingSlideTranslator), so repeated deferred passes re-pay LLM cost.
- Cell referencing is ≥5 coexisting mechanisms layered per-issue: (slide_id, role) keys (#166, FileState.find_cell); content anchors with id: > construct: > hash: precedence (#190, anchor_of sync_writeback.py:296) with a stored-row mirror row_anchor:317 that 'must stay in lockstep'; per-language positional ordinals for id-less localized cells (#198/#365, replace_idless_localized_*:514-609); Python object identity for anchor-keyed narratives (#403, replace_cell_body_obj:625, insert_after_cell:690) and for the run-scoped translation cache keyed by id(cell) (sync_apply.py:2869); occurrence-qualified vo_anchor tokens for voiceover placement (anchor_primitives via voiceover_tools).
- split.py is the one component with a declared algebraic contract — byte-identical round trip unify(split(x))==x, Hypothesis-pinned (split.py:6-17) — and it works precisely because split refuses to normalize anything (id minting explicitly excluded, :432-458; preamble-code left broken-but-byte-identical, :407-429). It also splits/unifies companions in lockstep atomically (:44-58, 306-337, 714-740). The sync projection inverted this choice (extract DOES normalize) and inherited the compensation stack as the price.
- Worktree-state caveat: at this worktree's HEAD (3980f9ab) apply refuses ALL companion-aware pairs (sync_apply.py:363-377 'companion write-back is not implemented'); Phase 2 (PR #515, commits 52cb8a3b/c1042b16, merged to origin/master 2026-07-01) replaces that guard. Even with Phase 2, cold-start bootstrap (mint/adopt/reconcile) over a separated pair is refused — the author must run assign-ids first.

## Pain points

- Canonical-form-by-implementation (accidental): the separated-pair fixed point is 'whatever extract_pair_text emits, re-projected through inline_pair_text'. Three compensations (both-halves re-extract, watermark-over-disk-re-projection, representation markers with cold-start demotion) exist solely because the projection pair isn't an involution. A declared normal form (or a projection that round-trips by construction, as split/unify does) would delete all three.
- Identity mechanism sprawl (mostly accidental): five referencing schemes — (slide_id,role), content anchor id:>construct:>hash:, positional ordinal, object identity, vo_anchor — each with its own drift-refusal code path (e.g. replace_idless_localized_body returns False on role drift; _stamp cold-adopt returns 0 on stream-length drift; replace_cell_body_obj returns False when the instance vanished). Each was the minimal fix for one issue; together they mean a cell's 'name' depends on which pass is looking at it. Essential core is real (legacy id-less cells + translated bodies can't hash-match cross-language), but at most two schemes (stable id + content anchor) look inherent.
- Multiple sources of truth with fragile coupling: 4 files + watermark cache + git HEAD + optional ledger + representation marker. anchor_of/row_anchor and the walk pairs ('materialize walks enumerate a superset of what the execute walks translate — a miss never happens', sync_apply.py:2384-2386) are lockstep-by-comment, not by construction. Hash canonicalization changes require a manual WATERMARK_HASH_VERSION bump (sync_writeback.py:255-258).
- TOCTOU / staleness windows: one apply reads the same files up to three times (plan build; _flush_companion_writeback dirty-diff via _read_or_none; _inlined_from_disk for the watermark). Apply trusts plan.projected_*_text without re-validating disk, so an external edit between plan and flush is overwritten for separated pairs; a concurrent writer between flush and watermark-record poisons the recorded baseline.
- Residual non-atomicity: atomic_write_all's replace phase is serial (crash window between os.replace calls leaves e.g. new deck + old companion); companion deletions and other-layout pruning happen after the batch — a crash there resurrects deleted narration on the next inline; two call sites still use the plain non-atomic FileState.flush (sync_apply.py:2026 cold-adopt id stamping writes ONE half only; web/studio/service.py:500).
- Deferred-pass cost loop: a separated pair's partial watermark advance is unimplemented ('the partial-advance path is not modeled over inlined text in Phase 2'), so any deferral holds the WHOLE watermark and next run re-baselines from projected git-HEAD; combined with the uncached sync-path translator, a deck with one persistently-deferred cell re-translates and re-diffs everything each run — plausibly a driver of the 'agents get stuck / fall back to manual' complaint.
- Refusal-heavy UX (partly essential, partly accidental): MIXED, CROSS_LANGUAGE, orphaned for_slide, unbootstrapped separated pairs, drifted positional streams, both-direction moves, ambiguous regions without --llm-recover all refuse/defer with instructions to run OTHER commands first (normalize, voiceover extract/inline, assign-ids, --baseline). Safe by design ('a wrong id is worse than a deferred one') but the recovery path is spread across many verbs and flags — the opacity the maintainer reports.
- Byte-faithfulness tax (essential given git-diff-friendly goals, but pervasive): trailing-blank/terminal-newline bookkeeping (_place_inserted, normalize_displaced_last, separator_blanks majority-vote heuristic with a documented non-uniform-deck limitation, sync_writeback.py:653-672) and reflow-insensitive markdown hashing (normalize_for_hash, :117-238) exist because the representation is raw lines, not a parsed model with a deterministic printer.
- Scale: the sync engine proper is ~19.4k lines across 20 modules, dominated by sync_plan.py (4829) and sync_apply.py (4399); sync_apply's docstrings are largely cross-references to 15+ issue numbers (#166 #190 #198 #202 #216 #228 #269 #289 #365 #403 #429 #438 #440 #448 #458 #501), i.e. the design is the union of its bug-fix history.
- Prompt/caching key micro-complexity (accidental): translation cache versions use hand-rolled namespaces ('v1:g<sha>' vs 'v1:gm<fp>' with \x1e/\x1f separators to avoid collisions, sync_translate.py:150-183) plus model and prog_lang folding in CachingSlideTranslator — three ad-hoc layers doing what one structured cache key would.
- _SYNC_ROLE_TAGS is duplicated from sync_plan into sync_writeback to break an import cycle (sync_writeback.py:43-45) — a small but telling layering inversion: the byte-level write module owns role semantics the planner also needs.
