## Summary

Read all four sync design docs plus the successor (sync-agent-toolkit-redesign.md) and current code (sync_plan.py) to verify what shipped. The four docs form a strict chronological repair chain, each fixing structural blind spots of the previous layer. (1) single-language-authoring-sync.md (#166, 2026-05-31) defines the intended workflow — author edits ONE language, one command mirrors the other, ids minted as a byproduct — and its core mechanism: a per-pair "structural watermark" in sqlite (ordered per-lang rows of (position, slide_id, role, content_hash)) as the change baseline, explicitly REJECTING git HEAD as "fragile to commit-before-sync". Cells are keyed by (slide_id, role); classification is add/edit/move/remove/conflict with per-cell direction. (2) sync-content-anchor-identity.md (#190, 2026-06-01) fixes the fact that the keyed classifier structurally cannot see ~66% of real cells (language-neutral code, id-less localized cells — two separate gates: role_of() and the lang filter in ordered_sync_cells), by adding a SECOND, parallel identity system: content-derived anchors (hand-id > AST construct > sha256) stored in a widened watermark, plus a structural-signature pass in sync_code.py — three alignment mechanisms now coexist in one engine. (3) sync-plan-resolve-apply.md (#216) re-architects the plan/apply boundary because apply_plan had become a second decision-maker (dry-run promised "N add", apply refused all; id-carrying cold pairs silently DOUBLED both decks with errors=[]); introduces dispositions (apply/refuse/conflict/pending) and the faithfulness contract (dry-run = pure stages 1+2a, byte-shared with the writing run). (4) sync-git-as-baseline.md (#366, 2026-06-21) is an exploration note admitting the watermark — the load-bearing choice of doc 1 — is "out-of-band stateful coupling ... a separate store that can rot" and the root trigger of the #363–#366 bug batch; it recommends an in-repo sync-state file (mechanism c), which was NEVER built — the successor agent-toolkit redesign (settled 2026-06-22, shipped clm 1.16) instead demoted the watermark: git HEAD is the default baseline for the read verbs (report/verify) and the watermark is only an opt-in accelerator for apply/autopilot, i.e. roughly the alternative doc 1 rejected.

The evolution is thus: v1 pair-walker (UPDATE-only over shared (slide_id,role), global source-lang inference, deleted 2026-06-08) → watermark+classifier engine (#166) → +content-anchor parallel pass (#190) → resolve-then-apply purity split (#216) → watermark demoted to cache, git-HEAD-first for reads (#366/#440). Each step is ADDITIVE ("without changing that engine's contracts" is stated as a virtue in doc 2), so the current engine stacks: keyed classifier + anchor diff + structural-signature rebuild + narrative-identity map (#403) + voiceover projection (#501) + ledger overlay (#448) + cold-start mint/adopt/reconcile whole-plan short-circuits — all feeding one SyncPlan. Current code (sync_plan.py:3926) still resolves baseline as baseline_from → baseline_ref → watermark → git HEAD → none inside the engine; the git-HEAD default lives at the CLI verb layer.

Gaps between promise and reality are extensive and mostly self-documented in later docs: #166 promised atomic per-proposal writes (doc 2 §1 admits partial flushes on error, fixed only in Phase 0 temp-swap, with a residual two-file race); #166's plan claimed to describe what apply does (doc 3 §1 disproves it); #166 phases 1–5 silently dropped every code cell and untagged markdown (doc 1 Phase 6 admits "translated headings over stale code ... incoherent"); the watermark's raison d'être ("defeats commit-before-sync", git only a fallback) was inverted within 3 weeks when the "active bug cluster is almost entirely watermark bugs" (agent-toolkit §1). None of the four docs mention separated voiceover companion FILES (voiceover_*.de/.en.py) — "voiceover" appears only as a cell role inside the deck — which matches the maintainer's discovery that many commands ignored sidecars entirely; the projection fix (#501) was bolted onto build_sync_plan in 2026-07. The docs repeatedly acknowledge heuristic residue: the "id-less heuristic", sha256-fallback churn ("honest residual"), mtime tiebreaks, direction-from-narrative-proposals failing on code-only edits, the Phase-1b position-space trap (recording more cells manufactures spurious moves), and the in_sync-banking saga that took four adversarial-review rounds to settle.

## Key findings

- ARCHITECTURE (doc 1, #166): engine = structural watermark (sqlite table sync_watermarks in clm-llm.sqlite: ordered per-lang rows (position, slide_id, role, content_hash), advanced only on clean apply) + classifier producing typed proposals add/edit/move/remove/conflict with per-cell direction; adds detected by ABSENCE of slide_id ('id-less-as-new'); ids minted EN-authority (translate EN first, slug from EN heading, stamp both siblings, sticky once persisted); default writes to working tree, git diff is the review surface.
- REJECTED ALTERNATIVES (doc 1 §3.1-3.2): git HEAD as sole baseline ('fragile to commit-before-sync'); merge-base ('wrong frame'); content-similarity move detection (bias toward 'add' because wrong fusion corrupts the cross-language join); in-file conflict markers (corrupt percent format). REJECTED (doc 2 §3-4): blanket id-on-every-cell (a ~13,000-id write, 66.5% of all cells — 'the cure becomes the disease'); SyncCache translation memo as identity; git HEAD~1 inter-commit diffs (squash merges collapse them).
- EVOLUTION 1 — resolve-then-apply (doc 3) replaced 'apply as second decision-maker': apply_plan re-ran structural logic the plan never captured (both-directions id-less refusal at sync_apply.py:710-718; the id-CARRYING path was unguarded at :695-705 → silent doubling of both decks with errors=[]; #216 fresh decks could not bootstrap). Fix = dispositions (apply/refuse/conflict/pending) decided in pure stage 2a; LLM materialize (2b) split from mechanical execute (3); faithfulness contract: dry-run = stages 1+2a byte-shared with the writing run.
- EVOLUTION 2 — content-anchor identity (doc 2) replaced nothing; it ADDED a parallel identity system because the keyed classifier structurally cannot see id-less or language-neutral cells (two gates: sync_writeback.role_of() returns None; ordered_sync_cells drops lang!=expected at sync_plan.py:211 — a neutral cell is dropped from BOTH lists). Anchor = hand-id > AST construct > sha256(body), stored in DB (watermark 'construct' column + 'shared' partition), never written to files except the deterministic def-my-fun id-migration correction.
- EVOLUTION 3 — baselines moved to git-HEAD because the watermark became the dominant bug source: sync-git-as-baseline.md names it 'out-of-band stateful coupling... nothing enforces lockstep' (edit both halves + commit without sync → stale watermark → later run errors against a fine pair; root trigger of #363-#366). Its recommended mechanism (c) (in-repo .clm-sync-state file with serialized BaselineBundle) was NEVER built; the successor sync-agent-toolkit-redesign.md (settled 2026-06-22) instead demoted the watermark: report/verify default to git HEAD, watermark = opt-in accelerator for apply/autopilot only, 'rebuildable, never the authority' — approximately the alternative doc 1 rejected. Enabler: since #289 P1 all baseline consumers read one shape (BaselineBundle) producible from watermark OR any git ref through the same chokepoints.
- STATED INVARIANTS: (a) no-silent-no-op — every run reports the baseline used, distinguishes 'in sync' from 'no baseline'; (b) per-proposal atomicity across both decks, one flush per deck; (c) watermark advances only over clean/reconciled state (later refined: per-cell partial advance only for content-only issue-free passes, preserve-keys must exist in BOTH decks, tag-only holds scoped via TagHold); (d) ids sticky once persisted, never regenerated without --force; (e) EN-authority: de_id == en_id by construction (_slide_ids_pair); (f) byte-preserving split/unify round-trip incl. header + trailing blanks; (g) unify byte-identity of language-neutral shared cells across halves; (h) corpus no-op invariant: clean re-run = zero bytes written, zero LLM calls (212-deck harness, 81/212 no-op, 0 violations held through every phase); (i) dry-run/apply structural-disposition parity, sole permitted divergence = pending→blocked; (j) classifier is pure (no LLM in sync_plan.py); (k) neutral/id-less code never minted a slide_id; (l) all LLM outputs hard-validated (validate_alignment, validate_correspondence) with safe-abort-to-deferral.
- GAP 1 — atomicity: doc 1 promised per-proposal atomic writes; doc 2 §1/§11 admits apply flushed partial writes on any mid-pass error (sync_apply.py:225-226 unconditional flush; guard at :240 protected only the watermark). Fixed by Phase-0 buffered temp-swap, residual race between the two os.replace calls remains; doc 1 Phase 4 Part 1 further admits 'a structural change is flushed to the working tree even on an erroring pass'.
- GAP 2 — scope: doc 1 phases 1-5 shipped an 'engine' that dropped every code cell and every untagged/alt markdown cell; Phase 6 admits a real editing pass produced 'translated headings over stale code' — incoherent output. Doc 2 then measured the blind spot: 8,014 cells exposed to silent drop (item 2) + 1,702 exposed to needless re-translation (item 3) out of 19,787.
- GAP 3 — the plan lied: doc 3's symptom table — dry-run printed 'N add' while the writing run deferred everything (exit 1 vs 2); cold-start mismatched-id/half-id'd pairs silently doubled both decks; #216 could not bootstrap. Also #225 (doc 3 §12): a COMMITTED id-less pair resolved to a git-HEAD baseline and silently doubled, because the keyed engine's baseline is only meaningful when the halves share ≥1 slide_id — patched by the _pair_is_unbootstrapped demotion heuristic.
- GAP 4 — principle reinterpretation: doc 1 §3.2 forbids similarity-guessing cross-language identity; doc 3 Q-B introduces LLM correspondence verification for cold-start minting and declares it 'an explicit semantic verification is not a blind structural guess' — extending-not-breaking by fiat.
- GAP 5 — voiceover blindness: none of the four docs models separated voiceover companion FILES; 'voiceover' is only a cell role inside the deck. Issue #501 (a month after doc 1) found clm slides sync entirely blind to voiceover_*.de/.en.py; the fix is an in-memory projection (project_pair) bolted into build_sync_plan (sync_plan.py:3973-3987) with its own refusal path — a fifth identity/alignment layer.
- GAP 6 — purity boundary incomplete by decision: doc 3 status block records the id-migration recoverer 'stays inline by decision, not omission' because predicting its stuck-gate would require simulating the deterministic tier ('a lock-step contract bought for zero behavioral gain') — i.e. the resolve/apply architecture could not be applied uniformly to its own components.
- AD-HOC SIDE-PATHS: mint, adopt, and reconcile are each WHOLE-PLAN SHORT-CIRCUITS in apply_plan (doc 3 §12-13), bypassing the normal proposal walk because per-cell processing 'can cross-add a confirmed twin in both directions (doubling)', the partial-watermark advance 'tracks only one (slide_id, role) per proposal', and the id-less walk 'would double a deferred id-less suspect' — three enumerated reasons the general machinery cannot express these cases.
- WATERMARK-ADVANCE SEMANTICS were not derivable from a principled model: doc 1 Part 2b ('the deepest rabbit hole of the issue') took four adversarial-review rounds, one full design flip-flop on in_sync semantics (bank vs preserve), and found three genuine bugs including a phantom-add that re-created deleted slides; eligibility for partial advance is a conjunction of five conditions (content-only, real baseline, no errors, ≥1 deferral, ≥1 applied edit) plus a both-decks-presence guard plus TagHold scoping (#202).
- CURRENT CODE STATE: build_sync_plan (sync_plan.py:3912-3968) baseline priority is baseline_from → baseline_ref → watermark → git HEAD → none — the watermark still outranks git HEAD inside the engine; the 'git-HEAD default' shipped at the verb layer (report/verify pass no watermark). build_sync_plan's signature now takes 8 orthogonal knobs (watermark_cache, allow_git_fallback, provider_available, baseline_ref, baseline_from, detect_rename, ledger, + paths) and its docstring must explain four issue numbers (#216, #228, #440, #448, #403, #501) to describe one function.

## Pain points

- Layered accretion instead of consolidation: each design is explicitly 'additive... without changing that engine's contracts' (doc 2 §16), so the engine now runs at least five coexisting alignment mechanisms — keyed (slide_id,role) classifier, anchor-keyed diff (align_anchored), structural-signature rebuild (sync_code ('R'/'S'/'L'/'J') signatures), narrative-identity map (#403 occurrence-under-slide), and voiceover projection (#501) — reconciled 'at the proposal level' into one SyncPlan. Doc 2 §6 itself calls the anchor pass 'the missing component every naive approach assumes but none specifies'. This is ACCIDENTAL complexity: a single identity model designed up-front for all cell populations would not need parallel passes guarded by a corpus harness against double-emission.
- Identity is fragmented by cell population: narrative id'd cells key by (slide_id, role); neutral code by ordered content-hash sequence ('NOT an anchor map — a construct anchor isn't content-unique... collapses duplicates last-writer-wins', doc 2 §14 3a); localized id-less code by construct-anchor with Counter-dedup; id-less narratives by position-in-membership-widened-watermark guarded by body-hash (doc 1 Phase 6 retag note); sha256-fallback cells churn as remove+add on every edit ('honest residual'). Each population has its own matching rule, failure mode, and guard — the maintainer's 'cell referencing/alignment feels ad-hoc' is confirmed by the docs' own language.
- Position-space coupling (Phase 1b trap, doc 2 §14): move detection keys on position AMONG THE role_of!=None SUBSET, so merely recording more cells in the watermark 'shifts those positions and would manufacture spurious moves'; membership widening had to give widened rows a separate position space. Identity that changes when you observe more of the file is a designed-in fragility.
- The watermark was the central architectural bet of doc 1 (git HEAD explicitly rejected) and became the dominant bug class within 3 weeks (agent-toolkit §1: 'the active bug cluster is almost entirely watermark bugs' — #435 worktree key miss, #430, #364 stale hard-error, #366 commit coupling). The correction (demote to accelerator) happened at the CLI-verb layer while the engine still prefers the watermark internally (sync_plan.py:3926) — two baseline philosophies coexist in the shipped system.
- Decision authority leaked across the plan/apply boundary from the start and was only partially repatriated: resolve-then-apply moved refusals into 2a but the id-migration recoverer stays inline (LLM inside apply) by explicit decision, structural translate was materialized only after follow-up #289, and mint/adopt/reconcile bypass the walk entirely as whole-plan short-circuits with a 'whole actionable plan' gate — the general architecture cannot express its own three cold-start cases.
- Every safety property is enforced by after-the-fact guards rather than by construction: _flag_residual_duplicates / _flag_cross_deck_orphans error 'so the watermark can never advance over a corrupt state'; validate_alignment/validate_correspondence hard-gate LLM output; the corpus no-op harness pins the invariant empirically over 212 real decks. The docs' own history (11 confirmed issues in Phase 2 review, 6 in Phase 3, 4 adversarial rounds for Part 2b, phantom-add bug, silent-doubling bugs) shows the invariants are not structural consequences of the model — they are patrolled borders.
- ESSENTIAL complexity, fairly attributed: (a) a 3-way problem genuinely needs a base state to distinguish edit direction from conflict — some last-reconciled record (watermark, git ref, or ledger) is unavoidable; (b) cross-language identity has no byte-level signal for translated content, so either persistent ids, content anchors, or an LLM verifier is required; (c) the author-never-manages-ids requirement forces the tool to own identity minting; (d) byte-preserving split/unify and LLM nondeterminism force caching/stickiness. ACCIDENTAL: the sqlite out-of-band store (vs an in-repo baseline, considered and deferred); per-population matching rules; position-keyed moves; plan/apply decision split; the conditional partial-watermark-advance state machine; whole-plan short-circuits; the 4-file voiceover reality retrofitted via projection instead of being in the core data model.
- The docs are internally honest but the honesty is scattered: 'id-less heuristic' (doc 1 §5 pipeline), mtime tiebreak for divergence winner (doc 2 §7a), direction-from-narrative-proposals failing on code-only edits ('an author virtually always also touches a slide/voiceover' — an assumption, doc 1 Phase 6), 'deliberate reorders of byte-identical boilerplate are invisible to content-anchors (accepted)' (doc 2 §15), conflict UI shipped two-up instead of the designed three-up because the watermark stores hashes not bodies (doc 1 Phase 4 2a). An assessor should treat these as the designers' own catalogue of where the model is heuristic.
- Bug-prone hot spots by doc evidence: cold-start/bootstrap paths (doubling ×3 variants: id-less, half-id'd, mismatched-id, committed-unbootstrapped #225, partial-overlap #226/#228 — five successive patches on 'what if the pair shares no ids'); watermark advance/hold logic (Part 2b + #202 + phantom-add); anything both-directions; shared-cell divergence; and every seam where a new cell population was added to an engine designed for a narrower one.
