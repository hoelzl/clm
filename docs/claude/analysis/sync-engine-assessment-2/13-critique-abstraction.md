# Critique: neutral abstraction analysis

> Evidence slice of the 2026-07-01 sync-engine architecture assessment
> (`docs/claude/sync-engine-architecture-assessment-2.md`).
> Produced by an independent review agent; verify file:line citations against
> the current tree before reuse (tree at review time: `3980f9ab`, pre-#515).

## Position

The sync engine's identity/alignment model is ad-hoc by accretion — roughly ten locally-principled but globally uncoordinated identity mechanisms selected per cell by incidental metadata — and a single total per-cell identity (persisted ids on every cell, one generic 3-way record diff, one content-keyed committed baseline) would structurally eliminate the majority of the recorded bug classes; the #501 inline projection is the right idea (4 files as renditions of one document) executed at the wrong layer (a non-involutive text transform instead of a parsed document model).

## Direct answers

### Is the current design salvageable?

Conditionally — through the abstraction lens: the apply-side safety layer (resolve-then-apply staging, atomic error-gated flush, validated LLM tiers, fail-safes), split.py's round-trip discipline, the translator/judge, and the verb-group CLI are sound and worth keeping; but the identity/alignment layer is not salvageable by further patching, because most of sync_plan.py and large parts of sync_apply.py are compensation machinery for cells lacking total stable identity (occurrence classifiers, Tier-C retag variants, narrative asymmetry passes, membership roles, cold-start short-circuits, position-space re-indexing, reconcile_vo_ids). Salvage means replacing the identity core and the baseline store while retaining the write/safety perimeter and the behavioral oracles — a consolidation, not a greenfield rewrite.

### Are the abstractions correct?

No. The identity abstractions are ad-hoc by accretion: roughly ten coexisting mechanisms ((slide_id,role) keys, occurrence-under-slide implemented three divergent ways, content anchors with a live/stored lockstep contract, demoted-but-persisted vo_anchor tokens, for_slide with legacy fallback chains, four incompatible position spaces, lossy structural signatures, two different content-hash notions, path-keyed state, id(proposal) decision keys), selected per cell by incidental metadata and held consistent by prose 'must mirror exactly' contracts with recorded violations. ~90% of the issue history maps onto identity seams, not coding errors. A systematic model would make identity TOTAL and PERSISTED: the unit is the logical cell; every cell carries a tooling-minted, language-independent id (de==en by construction); the baseline records whole cells keyed by that id in one committed content-keyed store; one generic 3-way diff replaces the ten channels. That model structurally eliminates the cold-start matrix, the keyed-vs-occurrence routing seams (#403/#443), the position-space interaction bugs (#282/#285), and the recorded-channel drops (#198/#269/#289), while the genuinely essential residue — both-sided conflicts, translation-equivalence judging, baseline staleness — remains but in one place instead of ten. The #501 projection is the right idea (4 files as renditions of one document) done late and at the wrong layer: a non-involutive per-run text transform with a compensation stack, where a parsed document model with a declared canonical form (a cell attribute 'stored-in-companion') would make the projection a trivial serialization concern.

## Argument

VERDICT: AD-HOC, WITH A PRECISE MEANING. Each mechanism below is individually careful — documented, guarded, tested. What is missing is a global identity discipline: there is no single answer to "which cell is this?", and the answer a cell gets depends on which pass is looking and on incidental metadata (does it happen to carry slide_id? lang? a narrative tag? is it j2? which file is it stored in?). The bug history concentrates almost entirely at the seams between mechanisms, not inside them.

(1) MECHANISM ENUMERATION (verified in code, this worktree):

M1. (slide_id, role) keyed pairing — identifies id'd narrative markdown, id'd aux markdown, and localized id'd code (synthetic CODE_ROLE). Dispatch predicate: role_of, src/clm/slides/sync_writeback.py:55-89. Authoritative whenever both halves carry the id (EN-authority minting guarantees de_id==en_id). Note the layering tell: _SYNC_ROLE_TAGS is duplicated into the low-level write module "to keep this ... free of an import cycle" (sync_writeback.py:42-45) even though tags.py declares itself the single source of truth.

M2. Occurrence-under-slide (owning_slide_id, role, occ) — identifies ID-LESS narratives only; id-carrying narratives stay on M1 ("Only id-less narratives are anchor-managed", sync_plan.py:2591-2593). Chosen mid-implementation after the predecessor anchor "proved unstable as an identity" under sibling insertion (sync_plan.py:2558-2568). Implemented THREE times with divergent semantics: sync_plan._index_narratives_by_anchor (id-less only), sync_apply._narrative_keys_by_index (apply-side mirror), and reconcile_vo_ids._narrative_index which deliberately counts over ALL narratives ("does not filter by id-ness", reconcile_vo_ids.py:85-93). The ordinal is position-derived, so it shifts under mutation — apply must resolve targets from a pre-mutation snapshot by Python object identity (the shipped #403 Phase B mis-targeting bug).

M3. Content anchors id: > construct: > hash: — identifies id-less/neutral code for the #190 anchored diff and §9 id migration (anchor_of, sync_writeback.py:296-314) with a row-side twin row_anchor whose docstring says "The two must stay in lockstep" (sync_writeback.py:317-322). The hash branch self-invalidates when the cell is edited (identity derived from content changes with the content), forcing a second drift detector (Counter multiset) in sync_code.

M4. vo_anchor predecessor tokens (id:/fp:/tm:<val>#occ) — persisted into companion files; demoted from identity to placement-only; still recorded in the watermark and in one place read-then-discarded solely to keep a coverage gate satisfied (sync_plan.py:2686-2688).

M5. for_slide slide references in voiceover companions, with a 4-step legacy fallback chain including two historical title forms (voiceover_tools.py:1124-1137, 1344-1387). Erased and re-derived by the #501 projection.

M6. Positional/ordinal schemes — at least four incompatible position spaces: stored watermark positions (all non-j2 cells), legacy classifier positions (real-role cells only, re-indexed via enumerate because _resolve_duplicates compares ABSOLUTE positions — verified at sync_plan.py:1437-1448), per-language ordinals for id-less localized cells, and RawCell raw_index; plus j2-cells-matched-by-ordinal in sync_code. Alignment holds only when separate functions iterate "in lock-step (identical predicate/order)".

M7. Structural signatures — a separate alignment engine over lossy per-class tuples ('R',role)/('S',body)/('L',kind)/('J',) in sync_code.py:246-293, each arm justified by a distinct issue.

M8. Two content-hash notions of "same content" — hash_cell (drift: reflow-normalized markdown, byte-exact code, comment-token-dependent, WATERMARK_HASH_VERSION-versioned; sync_writeback.py:264-273) vs body_fingerprint (placement: drops ALL blank lines specifically to be invariant under extract's own \n{3,} squash; anchor_primitives.py:66-74). The placement layer and the drift layer answer "did this change?" differently by design, undocumented at the model level.

M9. File-level twin derivation — two parallel .de/.en derivation families in pairing.py (prefix-gated vs prefix-agnostic), with the voiceover_ prefix special-cased in three separate functions; the docstring admits the logic was previously copied across three modules.

M10. State/decision identity — the watermark keys deck identity by absolute resolved path strings (filesystem path as identity: broke under worktrees #435 and cwd-as-root #477); the ledger keys trust by content hash per (slide_id, role) or narrative occurrence; report item ids are re-minted per invocation from a 6-tuple with list-order dedup suffixes; and apply decisions are keyed by Python id(proposal), valid only against the exact plan object, with the "rebuild after any re-plan or every key misses" landmine documented in the code itself (verified, sync_apply.py:765-766). Plus four synthetic membership roles invented so the watermark can record cells the per-cell engine disowns (sync_plan.py:89-95).

(2) CONFLICT STRUCTURE. The critical property is that a cell's identity REGIME is selected by incidental metadata: an author adding or removing a slide_id moves a narrative between M1 and M2 — two different classifiers with different conflict semantics — and the two halves may DISAGREE on id-ness, which is exactly the #443 silent-drop configuration. A whole module (reconcile_vo_ids.py) exists solely to shepherd cells between the two regimes, with its own third occurrence semantics. Cross-mechanism interactions each need bespoke guards: move x retag (_MOVE_RETAG_BLOCKERS, a frozenset of 8 proposal kinds), move x neutral edit (#282), reorder x id-less drift (#285), id'd x id-less narrative asymmetry (four cooperating passes: divert, reconcile, alert, mass-add guard). Consistency across the live side, baseline side, and apply side is maintained by prose contracts, and the code records that these contracts have already been violated: _baseline_owning_by_position's docstring says it "must mirror owning_group *exactly*, or ... a one-sided edit is silently dropped" and then documents the bug where keying on the anchor alone left a narrative unpaired (verified, sync_plan.py:1403-1428).

(3) BUG CLASSIFICATION BY FAILED MECHANISM (~35 defects from the issue history):
- M1 key too coarse or inconsistently applied: #403 (multiple voiceovers per slide collapse onto one key), #443 (plan/apply matched bare slide_id while verify had the (slide_id,role) fix — the same invariant maintained independently in two places), #162 (per-language minting divergence), duplicate-id heuristic resolution (sync_plan._resolve_duplicates).
- Regime-boundary (M1<->M2/M6 seams, id-state asymmetry): #216/#225/#226 cold-start doubling chain (three successive re-fixes), #438 false needs_agent on clean decks, #365 id-less localized both-sided drift, the #10/#443 reconcile-cancels-adds drop.
- M6 position spaces: #282 move-x-edit clobber, #285 retag-under-reorder, the documented Phase-1b trap (recording MORE watermark cells manufactures spurious moves), #403 Phase B occurrence-shift mis-targeting.
- M8 hash canonicalization: #429 (rewrap reads as edit), #458 (comment token hard-coded, // decks excluded), plus the versioning landmine that any canonical-form change invalidates all persisted state.
- M10 path-as-identity: #435, #477, #374.
- Unrecorded channels (baseline records selected properties, not whole cells): #198 tags, #269 neutral bodies, #289 recorded-but-never-read tag map.
- Unmodeled file scope: #501, #360, #324, #443 (companions outside the document model).
- M4 edge cases: #242/#246/#247.
Ordinary implementation bugs (#377 JSON truncation, #281) are roughly 10% of the record. The failure mass sits on identity seams and identity-adjacent state — this is the empirical signature of a wrong abstraction, not sloppy coding.

(4) WHAT A SINGLE COHERENT MODEL LOOKS LIKE. Unit of identity: the LOGICAL CELL — not the slide group (too coarse; tag and body edits are per-cell) and not role-within-slide (that IS the #403 bug). The model: a bilingual deck is ONE logical document; each logical cell has (a) a stable, persisted, language-independent cell identity, (b) a language scope (localized -> two renditions, neutral -> one shared rendition), (c) an owning group, (d) a storage location (deck or companion — an attribute, not an identity regime). Identity must be TOTAL (every cell, always) and INDEPENDENT of position, content, and predecessor — the history shows every derived scheme (position M6, content M3/M8, predecessor M4, occurrence M2) failed precisely where its derivation assumption broke, while persisted slide_id is the one mechanism that never generated a bug class by itself. Minimal on-disk representation: extend the existing slide_id syntax to a per-cell id on every cell in both halves (id'd narratives keep slide_id; additional cells get slide_id plus a mandatory intra-slide discriminator, or plain per-cell ids), minted EN-authority by tooling exactly as today. The recorded rejection of ids-everywhere (#190: "~13k-stamp churn, the cure becomes the disease") is substantially stale: extract's _ensure_slide_ids ALREADY mass-mints ids onto id-less content cells in the separated layout (voiceover_tools.py:689), the split format is now agent-authored (the maintainer's premise), and the design docs' own reopen trigger ("verifier pain") has arguably fired. With total identity, the engine collapses to: one pairing function; one baseline record per cell_id (content hash per language, tags, position, location — the WHOLE cell, so no per-channel enumeration); one generic 3-way diff (edit=hash delta on one side, conflict=both sides, move=position delta under same id, add/remove=id set difference, retag=tag delta); one committed content-keyed baseline store (the ledger direction, keyed by cell_id, not path, not position).

(5) BUGS STRUCTURALLY IMPOSSIBLE vs REMAINING under that model. Impossible by construction: the entire cold-start refuse/mint/adopt/reconcile matrix (#216/#225/#226/#438 — after a one-time migration there is no id-less population; cold start becomes the migration tool); #403 and #443 (each narrative owns a unique id; no keyed-vs-occurrence routing seam exists); #365 (no id-less localized cells); #282/#285 and the Phase-1b trap (position is never identity-bearing); the occurrence-shift mis-targeting class; reconcile_vo_ids and the four narrative-asymmetry passes (no dual regimes); the four synthetic membership roles and TagHold dual identity; id(proposal) staleness (value keys become trivially available); #198/#269/#289-class channel drops IF the baseline records whole cells and the diff is generic over the record. Remaining (essential): both-sided edits are genuine conflicts needing a policy or a human; translation equivalence still needs an LLM judge (identity says WHICH cells pair, never whether their contents agree); a baseline is still required for direction (but one store suffices); hash canonicalization discipline (#429/#458 class, reduced because hashes become change-detection only); duplicate ids from author copy-paste (becomes a crisp validation error instead of a heuristic); LLM I/O robustness (#377); byte-preserving round-trip discipline.

#501 PROJECTION ASSESSMENT: right idea, wrong layer, and late. The conceptual move — the 4 files are renditions of one logical document — is exactly correct, and the projection was the best locally-available patch given the engine as-is. But three properties mark it as a symptom rather than the fix. First, it is a per-run TEXT transform, not a model: companion identity (for_slide space) is erased and narrative occurrence keys are re-derived over the projection each run, so companion cells still have no identity in the engine (verified: project_pair, sync_companion.py:156-208, inlines and strips). Second, inline-then-extract is not an involution — extract mints ids, stamps for_slide/vo_anchor, squashes blank lines — so the canonical form is defined operationally by extract's implementation, and the compensation stack (re-extract both halves, watermark over inline(disk), representation marker with cold-start demotion) is the ongoing price; contrast split.py, which declares a byte-identical round-trip contract and needs no compensation. Third, the recorded reason for rejecting first-class companion cells — it would add "a second cell-identity model" — is itself the indictment: under a total per-cell identity, a companion cell is just a cell whose storage location differs, and the projection degenerates into a serialization concern. The projection is defensible triage; it is also the clearest single piece of evidence that the document model ("two text files") is narrower than the product ("one bilingual deck with narration renditions").

## Concrete plan

Framed as the abstraction-consolidation path (each phase independently shippable, keeps the strong apply-safety layer):

PHASE 0 — Decide and spec the identity model (~1 week, mostly design + adversarial review against real code, which history shows is mandatory). Unit = logical cell; on-disk = persisted per-cell id reusing the slide_id header syntax (id'd narratives unchanged; other cells gain ids), EN-authority minting, de==en by construction. Spec a DECLARED canonical form for the separated layout (what extract emits), with a round-trip contract like split.py's Hypothesis-pinned byte identity. Decide the companion model: companion cell = logical cell with location=companion; for_slide becomes derived metadata.

PHASE 1 — One-time corpus migration (~1 week incl. dogfooding on PythonCourses). Build `clm slides migrate-ids`: stamp ids on every cell of every pair using the existing machinery (assign-ids, extract's _ensure_slide_ids, positional SlidePair alignment) gated by the existing CorrespondenceVerifier for ambiguous pairs. This is the existing cold-start machinery run ONCE, then retired from the hot path. Add a hard validate rule: no id-less cell in a split pair.

PHASE 2 — Unified diff core (~2-3 weeks). New, small module: one pairing function (cell_id join), one baseline record per cell (full cell: per-lang body hash, tags, position, location), one generic 3-way diff producing typed proposals with per-kind dataclasses (not a union-of-None-fields). Delete, as they become unreachable: the occurrence classifier, the id-less localized classifiers (Tier C both variants), the narrative asymmetry passes, membership roles, the mint/adopt/reconcile short-circuits, reconcile_vo_ids, the plan-patching passes. Keep and rewire: the apply pipeline's resolve-then-apply staging, atomic flush, fail-safes (now cheaper — parity checks over id-joined cells), LLM validation gates, translator, judge.

PHASE 3 — One baseline store (~1 week). Committed, content+cell_id-keyed (the #448 ledger direction, which is already "the watermark done right"); sqlite demoted to a pure rebuildable cache or deleted; path keys gone. Report/task/accept/apply all read the same store — one baseline regime for the whole loop; stable value keys (cell_id, kind) replace id(proposal) and re-minted item ids.

PHASE 4 — Document model for the 4 files (~1-2 weeks). Parse once into a logical deck (cells with language scope + location); deck.de/.en and companion.de/.en become serializations with the declared canonical form; the #501 projection and its compensation stack (representation marker, both-halves re-extract, watermark-over-disk-reprojection) are deleted.

TEST STRATEGY: keep the behavioral oracles (corpus no-op, corpus mutation — extend to DE-side and both-sided mutations; move them into CI on the committed synthetic corpus), the #269 _falsely_consistent predicate, and dry-run/apply parity; accept that most of the 24k-LOC white-box suite is implementation-coupled and will be retired with the code it pins. Add property-based round-trip and convergence (multi-round fixpoint) tests, which are absent today.

WHAT TO KEEP UNCONDITIONALLY: split.py and its round-trip contract; the atomic write utilities; sync_recover's validated bounded-LLM tiers; sync_translate; the walker UX; the verb-group CLI shape (report/verify/apply/task/accept) with its baseline defaults unified.

Rough total: 6-9 weeks of focused work, front-loaded by the Phase 0/1 decisions; Phases 2-4 each deletable-fallback (old engine remains until parity on the corpus oracles).

## Risks

- The ids-everywhere migration writes a large one-time diff across every course repo (the #190 objection); mitigated by tooling-owned minting and the fact that extract already mass-mints ids in the separated layout, but downstream repos and open PRs will churn once.
- Duplicate/hand-edited ids become the single point of failure for the whole model; needs a hard validate gate and collision-refusing writes (today's heuristic _resolve_duplicates becomes an error path).
- The 24k-LOC test suite is heavily white-box (15+ private imports, per-file watermark seeders); a consolidation invalidates much of it, and the behavioral oracles that must carry the transition are currently CI-invisible on the real corpus (slow-marked, maintainer-machine-only).
- The existing engine encodes ~4 weeks of adversarially-discovered edge cases (title groups, j2 barriers, trailing-blank bookkeeping, header asymmetry); a rewrite of the diff core can regress these unless the corpus mutation oracle is extended (DE-side, both-sided, companion mutations) BEFORE the cutover.
- Declaring a canonical extract form changes bytes of existing companions once; without a versioned normal form this repeats the WATERMARK_HASH_VERSION class of invalidation landmines.
- Essential ambiguity remains: both-sided edits, translation equivalence, and stale-baseline false conflicts do not disappear — if expectations are set at 'the rewrite ends conflicts', the effort will be judged a failure; the realistic win is deleting the seam-bug classes and the compensation machinery.
- Every prior design doc needed an adversarial correction pass on first contact with the real code; Phase 0 must budget for the same or the new model will ship with its own first-draft errors.
- Total identity depends on EVERY write path stamping ids (Studio, normalize, translate, manual edits); one unstamped path silently re-creates an id-less population and with it the regime-boundary bug class.
