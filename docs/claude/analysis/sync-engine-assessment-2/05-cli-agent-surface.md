## Summary

The user/agent-facing sync surface is very large and layered: under `clm slides sync` there are 7 verbs (report, verify, diagnose, apply, task, accept, autopilot) plus a 6-subcommand `baseline` group (show/bless/seed/establish/clear/prune), a legacy `clm slides watermark` alias group (list/clear/prune), plus adjacent `slides suggest-sync` and `slides reconcile-vo-ids`, and 2 read-only MCP tools — roughly 20 entry points and ~82 option slots (~33 distinct flag names). Autopilot alone carries 26 options and an ~80-line mutual-exclusion matrix (17+ explicitly rejected combinations), and it duplicates four toolkit verbs as flags (--dry-run≈report, --explain≈report --explain, --verify≈verify, --rebaseline≈baseline bless). The core agent loop (report → apply → task/accept → verify) is well-conceived on paper — a tiered report (mechanical/assisted/ambiguity), framed model tasks with named validators, and validated write-back — and the individual pieces (fail-closed excerpt resolution, multi-cell-body rejection, structural write gates) are carefully engineered. But the loop is stateful across invocations in ways only conventions hold together: report/task/accept default to a git-HEAD baseline while apply defaults to the watermark, so consecutive steps of the same loop can classify the same deck differently; item ids are re-minted per invocation from a 6-tuple of (kind, role, slide_id, direction, positions) and accept re-plans from scratch, so any intermediate change invalidates ids with "re-run report"; and a reconciled edit that isn't committed/blessed/ledger-recorded re-surfaces as a both-edited *conflict* on the next git-HEAD report (the docs themselves admit "most early-baseline conflicts are false").

State outside the 4 deck files is substantial and overlapping: (1) the watermark — per-cell rows in a project-local SQLite keyed by absolute (de_path, en_path) path strings — whose staleness produces false conflicts and has spawned SIX repair mechanisms (auto-heal, --rebaseline, baseline bless, watermark clear, watermark prune, --no-cache/--no-watermark) plus rename-detection, --baseline-from, worktree keying (#435) and project-root cache discovery (#477) as patches for its path-keying; (2) the committed per-topic ledger (.clm/sync-ledger.json), content-hash keyed and fail-safe when stale, but with six distinct write paths (apply --ledger, autopilot --ledger, accept --record, bless --ledger, seed, establish) each with different gating/provenance rules and an unlocked read-modify-write; (3) git itself as the default baseline. The ledger is effectively the watermark done right (content-keyed, committed, fail-safe-stale) yet was added as an overlay rather than a replacement, so both must be maintained and a `seed` command exists purely to bridge them. The word "baseline" means three different things (a git-ref flag, the command group that manages the watermark store, and the report's baseline_source field).

Verdict on this slice: this is an accretion, not a focused toolset. The essential complexity — per-cell bidirectional direction inference needs a 3-way baseline; both-side edits are genuine conflicts; translation needs a model; 4-file atomicity — justifies perhaps 3-4 verbs and one baseline mechanism. What is accidental: the path-keyed SQLite watermark and its repair industry; per-verb baseline-default divergence inside one loop; three parallel cell-addressing schemes (sync-index, localized-index, (slide_id,role) key) whose correctness depends on documented call-ordering invariants; a 14-label kind vocabulary projected through tiers, frameable-subsets, and code-role subsets that must stay mirrored across three modules by comment; the legacy autopilot codepath maintained in parallel rather than being a scripted composition of the toolkit verbs; and the JSON envelope carrying both the blessed `report` block and a deprecated flat `plan` block. A minimal sufficient surface would be: `report` (one baseline rule, ledger-aware by default), `apply` (mechanical + accepts validated answers, same baseline rule, conflict policy as an option), `verify`, and one committed content-keyed baseline store — with autopilot as a thin loop over those verbs rather than a second engine entry point.

## Key findings

- Surface size: `clm slides sync` is a verb group with 7 verbs (report/verify/diagnose/apply/task/accept/autopilot) + a `baseline` subgroup with 6 subcommands (show/bless/seed/establish/clear/prune) + legacy `clm slides watermark` alias (list/clear/prune) + `slides suggest-sync` + `slides reconcile-vo-ids` = ~18-20 entry points; option slots per verb: report 8 (+1 no-op flag), verify 1, diagnose 2, apply 9, task 7, accept 9, autopilot 26, baseline group 19 — ~82 total, ~33 distinct flag names.
- Baseline defaults diverge WITHIN the canonical loop: report/task/accept default to git-HEAD (`use_wm = use_watermark or baseline_ref is not None`, sync.py:1817; task sync.py:2648-2653; accept sync.py:2856-2859) while `apply` defaults watermark ON (`--use-watermark/--no-watermark` default=True, sync.py:2084-2088) and autopilot always uses the watermark — so consecutive steps of one loop can classify the same deck differently, and the same flag has different polarity per verb (plain opt-in flag on report vs on-by-default toggle on apply).
- provider_available is hardcoded differently per verb over the same plan function: report/task/accept pass True ('the agent is the verifier', sync.py:1824, 2661, 2867), apply passes False so cold pairs classify as `refuse` residue (sync.py:2310-2318), autopilot computes it from key presence (sync_autopilot.py:684-685) — a cold pair is a 'mint/adopt task candidate' in report but 'refuse' in apply output, shifting vocabulary mid-loop.
- Item identity is re-derived per invocation: ids minted from kind+slide_id/direction/positions with -2/-3 dedup suffixes (sync_report.py:495-531); `accept` re-builds the whole plan and re-matches the item to a proposal by a 6-tuple (kind, role, slide_id, direction, source_position, target_position), rejecting with 're-run report' on ≠1 match (sync_accept.py:148-176). Positions shift after any accept/apply, so a multi-item session is O(N) full re-plans and stale-id rejections are a designed-in failure mode.
- False-conflict trap in the loop: with the git-HEAD default baseline, a reconciled edit (apply/accept wrote the twin) that is not yet committed/blessed/ledger-recorded reads as 'edited on both halves since HEAD' on the next report — i.e. a tier-3 conflict. commands.md:2308-2312 concedes 'most early-baseline conflicts are false — a consistent bilingual edit'; the entire #448 ledger + #451 conflict-excerpt features exist to mitigate this baseline-scheme artifact.
- Watermark = per-cell rows in project-local SQLite (clm-llm.sqlite) keyed by absolute resolved (de_path, en_path) STRINGS (watermark.py:1-24; sync.py:762-768 canonicalization comment admits the same pair previously acquired two keys across surfaces). Staleness/orphaning has SIX repair mechanisms: --auto-heal (default-on, sync.py:1008-1044), legacy --rebaseline (sync.py:1071-1103), baseline bless (watermark.py:426-506), watermark clear/prune (watermark.py:233-392), --no-cache/--no-watermark; plus rename auto-detection, --baseline-from (sync.py:279-299), worktree main-checkout keying (#435) and project-root cache discovery (#477) as further patches on path-keying (commands.md:2004-2014).
- Ledger (.clm/sync-ledger.json, committed, per-topic) is content-hash keyed per (slide_id, role) + (owning_slide_id, role, occ) for id-less narratives, fail-safe when stale (hash mismatch → re-check), but has SIX write paths with distinct gates/provenance: apply --ledger (full-advance gate, sync.py:2251-2294), autopilot --ledger, accept --record (edit-kind only; other kinds print 'nothing recorded', sync.py:2917-2971), bless --ledger, baseline seed (oracle=assume), baseline establish (LLM-judged, the only model-bearing baseline verb, watermark.py:695-820). record_pair's load→update→save is explicitly unlocked (sync_ledger.py:420-424). The ledger duplicates the watermark's purpose with better properties; `seed` exists solely to bridge the two stores (sync_ledger.py:570-642).
- Ledger participation is inconsistent across the loop: report and apply accept --ledger, but `task` and `accept` build their plans WITHOUT a ledger (sync.py:2655-2665, 2861-2871) — so a `report --ledger` shows fewer items than `task` will frame for the same deck.
- Autopilot = the pre-redesign engine relocated, not decomposed: 26 options incl. 4 model backends (judge via --provider/--llm-model/--ollama-url/--llm-timeout, translator --translation-model, cold-pair verifier Haiku --verify-cold-pairs, recovery Opus --llm-recover/--recovery-model), ~17 mutually-exclusive combinations rejected across sync_autopilot.py:590-671, and 4 flags duplicating toolkit verbs (--dry-run/--explain/--verify/--rebaseline). Each of the 4 model clients independently degrades to None on a missing key with different downstream semantics (edits→errors, adds→defer, cold pairs→refuse, realign→left), so a run's outcome ('N deferred; M error(s); watermark held') requires knowing the tier system to interpret. Its name suggests the recommended autonomous path while the code calls it 'the legacy all-in-one / agent-less human's escape hatch' (sync_autopilot.py:1-18).
- Cell referencing uses THREE parallel schemes whose correctness rests on documented call-ordering invariants: the 'sync' index (sync-relevant cells), the 'localized' index (non-j2 lang cells), and (slide_id, role) key lookup — sync_report.py:88-96 states the _LOCALIZED_POSITION_ROLES set 'is only sound BECAUSE _enrich routes every keyed item through resolve_by_key and returns BEFORE the positional path runs'; issue items must be handled before the keyed branch (sync_report.py:363-381). The realign item uses a fourth convention (inline bytes, role 'neutral-code', sync_report.py:304-308).
- Kind/tier vocabulary sprawl: 11 proposal kinds (add/edit/retag/move/remove/conflict/rename/refuse/mint/adopt/reconcile) + synthetic kinds issue/realign/neutral-propagate = 14 labels, projected into 3 tiers via _MECHANICAL_KINDS/_ASSISTED_KINDS (sync_report.py:73-77), re-subsetted as _FRAMEABLE_KINDS (sync_task.py:86) and _CODE_EDIT_ROLES which 'mirrors sync_apply._resolve_edit' in TWO other modules by comment (sync_task.py:88-93, sync_accept.py:75-77); `diagnose` adds its own 9-label root-cause catalog (sync_diagnose.py:42-53).
- MCP surface is read-only and drifts from the CLI: only `slides_suggest_sync` (legacy bilingual single-file, tools.py:713-735) and `sync_report` (tools.py:758-823); no MCP task/accept/apply/verify, so an MCP-driven agent can diagnose but not act. handle_sync_report uses the watermark whenever the cache DB file exists (tools.py:801-815) while the CLI `report` verb defaults to git-HEAD ignoring the watermark — the docstring claims 'matching the CLI' but matches the pre-redesign behavior, so MCP and CLI reports can differ for the same deck.
- Voiceover-companion support is read-only in this tree: report/verify/diagnose inline companions in memory (#501 Phase 1; sync_verify.py:323-331, sync_report.py:250-254), and pointing sync at a voiceover_* file resolves its deck (sync.py:182-263), but commands.md:1790-1791 states 'Applying the reconciliation (the four-file write-back) is not yet implemented — apply on a separated pair reports the drift and writes nothing' — the loop dead-ends at apply for every separated-voiceover deck (the historical 'commands ignored sidecars entirely' bug class, partially fixed).
- Accept coupling is convention-only: commands.md:2154 requires the baseline flags on accept to 'match the report/task that framed the item', and --since 're-resolves the same timeframe — stable as long as HEAD has not moved' — a documented race across three separate process invocations with no token/handle tying them together.
- Genuinely good safety engineering exists at the write boundary: fail-closed excerpt resolution (sync_report.py:44-47), multi-cell-body smuggling rejection using the parser's own boundary predicate (sync_accept.py:229-247), strict_single translator preventing a co-drifted sibling from being silently overwritten with the wrong answer (sync_accept.py:185-219), structural_gate as a shared write-gate for watermark+ledger (sync_verify.py:138-179), auto-heal gated on a verified git-HEAD no-op (sync.py:1008-1044), unknown kinds defaulting to ambiguity never mechanical (sync_report.py:169-185).
- Footgun/no-op flags: report's --no-env-file is 'accepted for symmetry; report reads no .env' (sync.py:1982-1986); the _DefaultVerbGroup treats any unknown first token as a deck path for `report`, so a verb typo yields a 'path does not exist' error rather than 'unknown command' (sync.py:1731-1747); `--baseline` works over a directory for report/apply but is rejected over a directory in autopilot (sync.py:707-711 vs sync.py:1839-1854) — the same flag, different batch rules per verb.

## Pain points

- Baseline-scheme accidental complexity dominates: the path-keyed SQLite watermark generated an entire repair subsystem (auto-heal, rebaseline, bless, clear, prune, seed, --baseline-from, rename detection, worktree keying, project-root discovery, hint texts) — roughly a third of sync.py's 3035 lines is watermark lifecycle management rather than reconciliation.
- One loop, three baseline regimes: report/task/accept (git-HEAD), apply (watermark), autopilot (watermark+models) — the agent must mentally track which store each step consulted; mismatches surface as vanished item ids, phantom conflicts, or 'the deck changed since report; re-run'.
- The false-conflict cycle: reconcile → don't commit → next git-HEAD report shows the reconciliation as a both-edited conflict; escaping requires knowing about --ledger/--record/bless — three opt-in mechanisms an agent must discover from a 406-line sync-agents info topic.
- Watermark and ledger are two overlapping last-synced-state stores with opposite staleness behavior (watermark stale → false errors needing heal; ledger stale → safe re-check); maintaining both, plus `seed` to bridge them, is pure accretion — the ledger's design obsoletes the watermark but did not replace it.
- Item addressing is ad-hoc as suspected: identity = slugged 6-tuple + list-order dedup suffix, resolved through 3 position/key schemes whose soundness depends on comment-enforced call ordering; no stable handle survives a single write, forcing full re-plan between every task/accept.
- Cross-module invariants held by comments: _CODE_EDIT_ROLES mirrored in sync_task and sync_accept 'to match sync_apply._resolve_edit'; tier sets in sync_report must track proposal kinds minted in sync_plan/sync_apply; realign items must mirror the apply-module detector 'so they can never disagree' — every new kind requires synchronized edits in 4+ files.
- Autopilot's 26 options and 17-way exclusion matrix are the visible cost of relocating rather than decomposing the legacy engine; it duplicates read verbs as flags and keeps --rebaseline alongside its own replacement (bless) and auto-heal.
- Exit-code and mode overloading: exit 2 means classifier error (report), structural corruption (verify), rejected answer (accept), refused pair or missing key (establish); the report JSON says mode:'dry-run'; the envelope carries both the blessed `report` block and the deprecated flat `plan` block plus walker/apply nulls.
- MCP/CLI divergence (watermark-if-exists vs git-HEAD default) and MCP's missing task/accept/apply verbs mean a non-shell agent sees a different report than the CLI and cannot complete the loop through MCP at all.
- Terminology collisions: 'baseline' = git-ref flag AND the watermark-store command group AND baseline_source values; 'verify' = structural verb AND autopilot's --verify flag AND --verify-cold-pairs (a completely different correspondence check); 'apply' = the tier-1 verb AND diagnose --apply AND apply_plan.
- Essential vs accidental split: essential = 3-way baseline for per-cell direction, real both-edit conflicts, model-mediated translation, 4-file (deck+companion) atomicity, cold-start identity minting; accidental = path-keyed watermark + repair industry, per-verb baseline divergence, re-minted item ids, dual trust stores, 14-kind vocabulary with mirrored subsets, parallel legacy codepaths (autopilot, suggest-sync, watermark group, flat plan block), read-only MCP twin.
