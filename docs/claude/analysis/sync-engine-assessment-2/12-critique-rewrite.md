# Critique: the case for a principled replacement

> Evidence slice of the 2026-07-01 sync-engine architecture assessment
> (`docs/claude/sync-engine-architecture-assessment-2.md`).
> Produced by an independent review agent; verify file:line citations against
> the current tree before reuse (tree at review time: `3980f9ab`, pre-#515).

## Position

The sync engine's failures are generative, not incidental — optional identity, out-of-band baseline state, and an unmodeled 4-file document each independently guarantee the observed bug classes — so the 9.2k-line plan/apply core should be replaced by a small principled engine (one canonical bilingual document, total block identity, lens-projected files, committed translation-memory ledger, per-block 3-way merge) built on the ~4.5k lines of perimeter the current system already got right.

## Direct answers

### Is the current design salvageable?

No for the core, yes for the perimeter. The plan/apply core (sync_plan.py 4,829 + sync_apply.py 4,399 lines, plus sync_code.py and the watermark lifecycle third of sync.py) is not worth salvaging: its two generative decisions — optional/heterogeneous cell identity and an out-of-band mutable baseline — are the traced root cause of ~80% of the ~35-defect history, and the engine's own design docs concede the repair arc 'has not converged... the signature of approximating an underspecified problem' (docs/claude/design/sync-baseline-storage-and-agent-direction.md:391-397). Patching cannot fix this because every patch is forced to add another identity scheme, another channel detector, or another state store — that is what the last four weeks empirically did (8 identity schemes, ~10 channels, 4 state-model iterations). What IS salvageable is substantial and high quality: the parsers (slide_parser, raw_cells), split.py's lens with its Hypothesis-pinned round-trip law, sync_translate, both validated LLM tiers in sync_recover, the committed ledger (sync_ledger.py — the correct baseline design, currently wrongly deployed as an overlay), sync_verify's structural gate, atomic_write_all, and above all the test-side invariants (no-op = zero bytes/zero LLM, propagate-or-alert, dry-run/apply parity) and the 212-deck corpus oracles. Roughly 4-4.5k of the ~19.4k lines seed the replacement directly; the invariants port as black-box acceptance tests.

### Are the abstractions correct?

No. Three abstractions are wrong, and each is independently falsified by the record. (1) Identity: the engine makes identity an optional, per-cell-negotiated property — ≥8 coexisting schemes ((slide_id,role) keys, occurrence-under-slide, content anchors, vo_anchor tokens, positional ordinals, baseline-position joins, construct slugs, id(proposal) object identity), selected by incidental metadata, with every pairwise interaction needing a bespoke guard. (2) State: the baseline is out-of-band mutable state (path-keyed sqlite watermark) that produced 10+ issues in 4 weeks and was redesigned four times; the committed content-keyed ledger that fixes it exists but was bolted on as an overlay instead of replacing it. (3) Scope: the product is a 4-file document but the engine models 2 files, with companions handled as an in-memory projection whose correctness rests on five baseline sources being projected identically. A systematic model: ONE canonical BilingualDoc (ordered blocks; each block shared-or-localized, narratives owned by a block, companion-vs-inline as a layout attribute); the 4 files as lens projections with declared round-trip laws (split.py already proves this pattern works in this repo); TOTAL identity — physical ids only for localized/narrative blocks (the population that genuinely needs cross-language names; shared blocks need byte-parity checks and sequence diff3, not names — dissolving the #190 '13k-id churn' objection, which conflated naming with alignment); and per-block sync state in the committed ledger in the translation-memory/XLIFF style (source-hash + target-state: verified/stale-de/stale-en/conflict). Reconciliation when both languages changed is then a per-block 3-way decision with no deck-level direction inference, no mtime tiebreaks, and no plan-patching passes — ambiguity becomes an agent task keyed by stable bid, not by Python object identity.

## 1. The engine's own artifacts concede the thesis

The strongest witness against the abstractions is the codebase itself. The maintainer-approved design doc docs/claude/design/sync-baseline-storage-and-agent-direction.md:391-397 (verified) states: "We have spent the #166 → #190 → #269 → #364/#365/#403 arc approximating semantic judgment with escalating deterministic heuristics, and the residue has *not* converged — each fix narrows the failure and exposes the next edge. That is the signature of approximating an underspecified problem." That is not a critic's framing; it is the project's own. The issue-history slice quantifies it: of ~35 distinct defects since the v2 engine shipped (2026-05-31), only ~11% are ordinary implementation bugs; ~80% fall in identity-model, unmodeled-scope, and state-management classes. When nine in ten bugs are the same three abstractions failing, the abstractions are the bug.

The prior assessment (docs/claude/sync-engine-architecture-assessment.md, verdict B "keep the engine") set explicit flip-to-replace conditions at line 120 (verified): (i) a body/structure-channel silent drop surviving the parity fail-safes on a clean pass; (iii) alert-rate evidence making the tool unusable. Both fired within three weeks. #443 was a body-channel silent drop ("decks already consistent", watermark advanced) that no fail-safe caught, invisible to the P0 completeness meta-test because it was a routing gap, not a channel gap — the failure mode moved from "uncovered matrix cell" to "unrepresented matrix axis", which is precisely what it looks like when the model, not the enumeration, is wrong. And the one real production reconcile (AZAV W10) flagged 56 conflicts + 17 edits of which 3 were genuine — a ~96% false-positive rate that forced the agent to hand-extract all 70 cells. By the previous assessment's own criteria, the verdict should now be (C).

## 2. The generative defect: identity is optional and heterogeneous

Every reader converged on the same count: at least seven or eight coexisting identity schemes — (slide_id, role) keys; four synthetic membership roles; occurrence-under-slide (src/clm/slides/sync_plan.py:2558-2568, verified: the docstring records that the predecessor anchor was tried as identity and demoted after proving "unstable as an identity"); vo_anchor predecessor tokens; content-hash anchors with uniqueness counters; AST construct slugs; baseline-position joins; and Python id(proposal) object identity for decisions. This is not diversity; it is the absence of an abstraction. Three exhibits, each verified in the tree:

- sync_plan.py:2686-2688: `_ = (de_reordered, en_reordered)` — the vo_anchor channel is read and its result **discarded**, purely so a coverage gate sees the channel "consumed" after its identity role was revoked. Code that exists to placate a meta-test about a demoted identity scheme is sediment, not design.
- sync_apply.py:765-766 (verified): decision maps are "Keyed by `id(proposal)`, so it MUST be built from the exact `plan` object passed to `apply_plan` (rebuild it after any auto-heal re-plan, or every key misses)" — with silent deferral as the failure mode. Cell identity is so unsettled that the API resorts to interpreter object identity, and the landmine is documented rather than fixed. It already bit once (#447, recorded in project memory).
- sync_plan.py:47-53 (verified): the classifier imports its core identity predicates (role_of, hash_cell, construct_of, row_anchor) **from sync_writeback, the apply-side write module** — identity has no owning module because it was never a first-class concept.

Downstream of optional identity, everything the readers flagged follows mechanically: ~10 parallel content channels each re-implementing the same 3-way state machine; four incompatible position spaces aligned by "lock-step iteration with identical predicates" comments; six post-passes that retract already-emitted proposals by id() (the plan is emitted, then patched — there is no single invariant defining a valid plan); and heuristics standing in where the model has no answer (mtime tiebreaks at sync_plan.py:1380-1400, a mass-add threshold of 3, exact slide_id-set rename matching). The bug history maps 1:1: #216/#225/#226 (three successive re-fixes of "what if the halves share no ids"), #365 (hash-only anchoring unsound under both-sided drift), #403 (keyed collapse of multiple narratives), #443 ((slide_id,role) vs slide_id keying fixed in verify but not plan/apply — the same invariant maintained independently in two places), #282/#285 (position-space interactions). Each is optional identity expressing itself.

## 3. The second defect: the baseline as out-of-band mutable state

The watermark — the load-bearing bet of the #166 design, which explicitly rejected git-HEAD — became "the source of nearly every active area:sync bug" (#440, three weeks later) and has now been through four models in four weeks: path-keyed sqlite → demoted to accelerator (clm 1.16) → re-promoted as a committed ledger overlay (#448) → plus a representation marker (#501). The end state is three overlapping trust stores (sqlite watermark, git-HEAD default, committed ledger) plus six ledger write paths, six watermark repair mechanisms, and a `seed` verb existing solely to bridge two of the stores. Roughly a third of the 3,035-line CLI module is watermark lifecycle, and ~15 test files carry their own divergent re-implementations of watermark seeding across three schema generations. Meanwhile the engine internally still prefers the watermark (sync_plan.py baseline chain) while the verbs default to git-HEAD — one loop, three baseline regimes, and the documented false-conflict cycle ("most early-baseline conflicts are false", commands.md) as the lived consequence. The ledger is the watermark done right — content-keyed, committed, fail-safe-stale — but deployed as an overlay it adds a store instead of removing two.

## 4. The third defect: the product is a 4-file document the engine never modeled

Separated voiceover companions were excluded from sync by a recorded 2026-06-02 decision; the compensating gate never got wired downstream; author-facing text actively said sync propagates companions; and a language shipped without narration while sync said "0 changes" (#501). The retrofit is an in-memory inline projection wrapped around the whole engine — correct only if the working tree AND all five baseline sources project identically, with a marker, ledger-space discipline, and a compensation stack for the fact that inline∘extract is not an involution (extract mints ids, stamps headers, squashes blank lines). The design doc's own words: modeling companions natively would require "a second cell-identity model" — i.e., the current identity model cannot express a second file kind. At this worktree's HEAD, apply simply refuses companion-aware pairs (sync_apply.py:369-376). Contrast split.py:1-16 (verified): a declared, Hypothesis-pinned byte-identical round-trip law — `unify(*split(x)) == x` — that handles companions in lockstep and has produced essentially zero bugs. The codebase already contains the correct pattern for multi-file views; the sync engine just wasn't built on it.

## 5. The replacement model, grounded only where theory genuinely fits

**Lenses / bidirectional transformations — fits, with in-repo proof.** The 4 files (deck.de.py, deck.en.py, voiceover_*.de.py, voiceover_*.en.py) are views of ONE canonical document. A `BilingualDoc` is an ordered sequence of blocks; each block is either `Shared(text)` (language-neutral code, j2) or `Localized(de, en)`; narratives are blocks with an `owner` bid and a `layout ∈ {inline, companion}` attribute. Parse = get; render = put; the round-trip laws are declared and property-tested exactly as split.py already does. Companion-vs-inline becomes a layout attribute of a block in one document — the entire #501 projection, representation marker, and 5-source baseline-projection threading disappear, because there is no second representation to keep in agreement.

**Translation-memory segment state (the l10n industry's 30-year-old solution) — fits.** For each localized block, the committed ledger records (de_hash, en_hash, state ∈ {verified(oracle), stale-en, stale-de, conflict}, provenance) — XLIFF's segment-state model. This is what the watermark's 5 partitions × rows/tags/anchors × 25 outcomes were groping toward. Direction is per-block from recorded state — no deck-level direction inference, no mtime tiebreaks, no align_anchored positional direction fallback.

**diff3 / sequence alignment — fits, and dissolves the census objection.** The #190 doc rejected universal ids because 66.5% of cells (≈13k) lack them (docs/claude/design/sync-content-anchor-identity.md:70-87, verified). But that argument conflates naming with alignment. Shared blocks are byte-identical across halves *by invariant* — their cross-language problem is a parity check, not a pairing problem; their within-language history is a sequence-diff problem, which diff3 over the block sequence solves without names, the way git merges unnamed lines. Only localized pairs and narratives need cross-language names — and the split format already stamps slide_ids on most of that population, agents author these files anyway (the maintainer says agent quality in the split format is high), and the residual stamp is a one-time adoption commit, not perpetual churn. Total identity where names are needed; sequence alignment where they aren't; nothing optional, nothing heterogeneous.

**Git as substrate — partially.** Base states are file contents at the last verified commit (already how git-ref baselines work); the ledger is committed and travels with the repo (#435/#477 path-keying bugs impossible by construction). **CRDT/OT — honestly does not fit**: there is no concurrent real-time editing; this is sequential 3-way reconciliation. **GumTree-style tree matching — only at adoption/recovery time** (matching un-id'd cells to mint bids); the existing construct_of + validated Opus recoverer already occupy exactly that niche and are kept there.

Reconciliation when both languages changed: per localized block, if either current hash matches a verified ledger pair → the other side is stale (directed task); if both drifted → conflict task with excerpts, resolvable by agent (or judge-verified equivalent → verified). Per shared block: parity or diff3; both-changed → conflict task. That is the whole classifier — one state machine, one identity, one position space, no post-passes.

## 6. Why the bar matters

The maintainer's competitor is Claude Code syncing decks manually, which mostly works. A 17.6k-line deterministic-heuristic engine cannot beat an agent by adding heuristics — the agent has semantic judgment; the engine's only durable advantages are (a) a trustworthy record of what was last verified, (b) mechanical propagation of the ~90% neutral/tag/structure cases at zero cost, (c) atomic multi-file writes, and (d) precise task framing. Those four things need perhaps 3k lines on top of the existing perimeter. Everything else the current engine does — direction inference, mass-add thresholds, plan-patching, three trust stores — is an attempt to out-guess the agent, and it is losing (96% FP noise, agents falling back to manual). The #440 pivot said exactly this ("put semantic judgment where it belongs") but only re-cut the surface; the replacement completes it structurally.

## 7. What the current engine gets right — and what a rewrite could lose

Honesty requires stating the counter-case. The write-boundary engineering is genuinely good: error-gated atomic flush, resolve-then-apply staging (no model calls after mutation begins), 6-gate validated LLM output with safe-abort, the no-op invariant (zero bytes, zero LLM), dry-run/apply parity, byte-faithfulness of untouched cells, refuse-over-guess discipline. The prior assessment's probes showed old fixes hold and severity trending down — it is *possible* the patched engine is a few bugs from stability. And the 24k-LOC test suite plus 189 issue references encode edge-case knowledge (title-slide identity, j2 barriers, duplicate slide_ids, terminal-newline cases) that a fresh implementation would otherwise re-learn through data loss. The mitigation is that all of these are policies and scenarios, not structures: they port as black-box acceptance tests (the corpus no-op backstop, the mutation oracle, the _falsely_consistent predicate, the verify gate) and the perimeter modules port as code. The corpus numbers also cap what "stability" could mean: only 81 of 212 real pairs can even reach a post-sync-clean state today (tests/slides/test_sync_corpus_noop.py:33-39, verified), with the regression floor set at 40 — the incumbent is not defending a working steady state.

## Concrete plan

**Target architecture (≈3-3.5k new lines + ≈4.5k reused, replacing ≈10-11k):**
- `bilingual_doc.py` (~600): BilingualDoc/Block model; parse (4 files → doc) and render (doc → 4 files) built on slide_parser/raw_cells/split.py machinery; declared normal form; Hypothesis round-trip laws (`render(parse(files)) == normalize(files)`, `parse(render(doc)) == doc`).
- `sync_state.py` (~400, evolved from sync_ledger.py's 967): the committed per-topic ledger as the ONLY store — per-localized-block (de_hash, en_hash, state, provenance, hash_version). Delete the sqlite watermark, the `baseline` group's watermark verbs, `seed`, auto-heal, rebaseline.
- `reconcile.py` (~600): the single per-block 3-way state machine (shared: parity/diff3; localized: ledger-state machine; structure: diff3 over the bid sequence). Pure, no LLM, no post-passes. Output: typed WorkPlan of {mechanical ops, tasks}, every item keyed by stable bid.
- `apply.py` (~400): apply mechanical ops to the doc in memory, render, atomic_write_all (≤4 files, one batch), update ledger. Fail-safes ported as black-box post-conditions over the rendered files.
- `cli` (~300): **one human command** `clm sync [PATH]` (applies mechanical, prints tasks; `--dry-run` for report-only); agent verbs: `clm sync --json`, `clm sync resolve <bid> --body-file -` (bid-keyed — survives re-plans by construction), `clm sync verify`, `clm sync adopt` (migration). 4-5 verbs, ~12 flags, versus today's ~20 entry points / ~82 option slots. Autopilot becomes a thin scripted loop over these verbs or is deleted.
- Reused as-is: sync_translate.py (433, add caching on the sync path), sync_recover.py (718, both validated LLM tiers, unchanged niche), sync_verify structural checks, sync_semantic judge, atomic_write_all, tags.py, pairing basics.

**Phases (agent-driven, differential-tested throughout):**
- **Phase 0 — spec hardening (1.5-2 wk, no engine code):** declare the canonical normal form (fixes the extract non-involution by decree — one-time normalize commit in course repos, hash_version bump); port the invariant suite black-box (no-op = zero bytes/zero LLM; propagate-or-alert via _falsely_consistent; dry-run parity; verify gate) so it can judge EITHER engine; get the corpus + mutation oracles running in CI on the committed synthetic corpus plus a scheduled real-corpus job. Exit criterion: the acceptance layer is engine-agnostic. Freeze the old engine (bug fixes only).
- **Phase 1 — model + read path (1.5-2 wk):** BilingualDoc lens with laws; `clm sync adopt` (stamp bids on the id-less localized/narrative residue — one commit per repo, agent-vetted; seed ledger via the existing establish/seed machinery); new `reconcile` powering a read-only `clm sync --dry-run`. Run differentially against old `report` over the 212-deck corpus and the mutation oracle; triage every disagreement.
- **Phase 2 — write path (1.5-2 wk):** apply + resolve + ledger update + ported fail-safes; companion write-back comes free via the lens (no projection). Exit: new engine ≥ old on detections, strictly fewer false positives, zero silent drops on the mutation oracle, all invariants green.
- **Phase 3 — cutover (1 wk):** `clm slides sync` verbs alias to the new engine with deprecation shims; delete sync_plan/sync_apply/sync_code/sync_companion-projection/watermark lifecycle/autopilot option matrix; update info topics (spec-files/commands/migration per the CRITICAL rule) and sync-agents.md (should shrink from 406 lines to ~80); MCP gets the same verbs (closing the read-only MCP gap).

Total ≈6-8 weeks of agent-driven effort — comparable to what the current engine consumed in June alone (30+ issues, 4 state-model iterations, with #501 Phase 3 and #448 parity still open). Retire ~10-11k of the ~19.4k sync-family lines and the majority of the 24k-LOC white-box test suite (scenarios recast black-box; the ~15 divergent watermark-seeding copies and 28-entry channel registry simply have no referent).

**Why rewrite #3 succeeds where #1 and #2 didn't:** (1) it changes the three decisions the defect record actually traces to — optional identity → total identity; out-of-band store → single committed ledger; 2-file model → 4-file canonical doc — whereas #2 changed everything EXCEPT those (it kept v1's optional identity and added the watermark); (2) the problem is now specified: the corpus, the mutation oracle, the invariant predicates, and 30 issues' worth of pinned scenarios are an executable spec that did not exist in May; (3) the format is settled (the split format stays), so the one-time adoption commit that #190 correctly priced out for a format in flux is now cheap and final; (4) the success bar is explicit and modest — beat manual Claude Code — which means the deterministic core should do LESS, not more: mechanical propagation + trustworthy state + atomic writes + bid-keyed tasks, with all semantic residue explicitly delegated to the agent.

## Risks

- Second-system risk: rewrite #3 could repeat the #166 trajectory (v2 also 'fixed the architecture'). Mitigation is structural: Phase 0 makes the acceptance layer engine-agnostic BEFORE any new engine code, and the old engine stays runnable until the differential harness shows superiority on the full corpus — but if Phase 1 disagreements are numerous and legitimate, the schedule doubles.
- Loss of encoded edge-case knowledge: the 189 issue references and 24k-LOC white-box suite contain landmines (title-slide identity, j2 barriers, duplicate slide_ids, terminal-newline handling, narrative mass-add pathologies) that black-box porting may miss; any missed one is a potential silent-drop regression — the worst failure class in this system's history.
- Normal-form declaration invalidates persisted hashes: declaring the canonical form (to fix inline∘extract non-involution) changes content hashes, requiring hash_version bumps and ledger/watermark re-seeding across course repos — a known landmine class (#429/#458) that has already produced two bugs when done piecemeal.
- Adoption-commit churn and downstream breakage: stamping bids on the id-less localized/narrative residue is a large one-time diff per course repo, and the CLI surface changes again ~5 weeks after downstream repos absorbed the 1.16 break; agent guidelines, MCP clients, and course-repo instructions all need synchronized updates (the #501 postmortem shows stale instructions actively cause shipped data loss).
- The semantic residue does not vanish: genuinely both-edited cells and never-verified legacy pairs (131 of 212 corpus pairs) become agent tasks; if ledger seeding is done cheaply (assume-rung instead of LLM-judged establish), the false-conflict noise that produced the 96% FP dogfooding experience partially survives, and the tool still loses to manual Claude on exactly the decks that matter.
- Convergence counter-factual: the prior assessment's probes showed fixes holding and severity trending down; it is possible the patched engine was 2-3 bugs from stability, in which case the rewrite spends 6-8 weeks to reach parity with what maintenance would have achieved — though the four-state-model churn and #501's threading cost argue marginal cost was rising, not falling.
- Opportunity cost and single-maintainer bandwidth: 6-8 agent-driven weeks with maintainer review on every phase competes with course production; a half-finished migration (two engines, two stores) is strictly worse than either endpoint, so the plan must be abandonable only at phase boundaries.
- diff3 over shared-block sequences has its own ambiguity edges (both-sided structural rearrangement of identical boilerplate) — the replacement must inherit the current engine's refuse-over-guess posture there, or it will introduce a new silent-reorder bug class the old engine's alert machinery already covers.
