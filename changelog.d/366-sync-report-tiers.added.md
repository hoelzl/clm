- **`clm slides sync --json` now carries a tiered `report` block — the agent contract (issue #366).** A coding agent both invokes sync and consumes its output, so the JSON gains a `report` block that partitions the engine's work into the **three tiers an agent acts on differently**: `mechanical` (the engine applies these deterministically with no model — move/remove/retag/verbatim neutral propagation — trust and ignore), `assisted` (a scoped model task the engine has already framed — translate a new slide, reconcile an id'd-cell edit, confirm a cold-pair correspondence — delegate to a cheap model or do it yourself), and `ambiguity` (the engine refuses to guess — a both-sided conflict or a structural issue — *your* judgement, stated as *what* is ambiguous, never a fabricated fix). Each item carries `kind`/`role`/`direction`/`slide_id`/`reason`/`severity` and 0-based `source_position`/`target_position`; the block exposes `is_clean`, `needs_model`, and `needs_agent`. An un-categorised future proposal kind defaults to `ambiguity`, never silently to `mechanical`. The existing flat `plan` block is unchanged (back-compatible). This is the second increment of the agent-first sync direction (after `--verify`); the `ReconciliationReport` Pydantic model is the stable contract the MCP sync surface can adopt once migrated off its legacy engine.
